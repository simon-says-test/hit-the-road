# Hit the Road — High-Level Design

Architecture/tech reference for the project. For intended gameplay behavior see [gameplay.md](gameplay.md); for contribution conventions see [CLAUDE.md](../CLAUDE.md) (not duplicated here).

**Status:** mid-rework, vertical endless scroller → closed-loop lap race, landing in phases (see gameplay.md's Status note). Phase 1 replaced the track/movement/camera model (closed-loop procedural track, heading-based steering, a following camera). Phase 2 brought rivals back as always-active 2D cars with track-following AI, full combat, and lap/position tracking; a follow-up pass (still within Phase 2's scope) added bidirectional rubber-banding, reintroduced rough/oil terrain hazards plus a new small-obstacle hazard type (placed once around the loop at race start), and added an `e2e/` Playwright suite (see Testing below) on top of the existing Vitest unit tests. Pickups are now implemented too: a chance-gated drop on every enemy kill, plus an independent timed standalone health/ammo crate spawn (`CRATE_SPAWN`, `PickupSystem`) — see Pickups below. `GameScene` also no longer owns every concern directly: lap/position tracking, HUD rendering, keyboard input, ram/hit damage math, and pickups have each been pulled out into their own `src/systems/` class (see Systems below), with `GameScene` left orchestrating spawn/lifecycle/collision-wiring. The minimap and the real lap/position HUD (a temporary debug readout stands in for now) are still pending.

## Tech stack

- **Language**: TypeScript 5.5 (`strict`, `noUnusedLocals`/`noUnusedParameters`, ES2020 target, `bundler` module resolution, no emit — Vite handles bundling)
- **Build/dev tool**: Vite 5.4 (`npm run dev` / `build` / `preview`)
- **Game framework**: Phaser 3.80 — rendering (Canvas/WebGL via `Phaser.AUTO`) and Arcade Physics for movement/collision
- No other runtime dependencies.

`src/entities/` and `src/config.ts` now hold the full gameplay layer: movement, weapons, enemy AI, pickups, and hazards. The Phaser-wrapper/pure-logic split established for `PlayerCar`/`playerPhysics` was carried through to `weapons.ts` (`WeaponController`, plus the shared side-gun mount-geometry functions `sideGunAngleDeg`/`sideGunMountPosition` — see Weapon visuals & in-world meter), `enemyBehaviors.ts` (`computeTargetHeading`/`turnTowardHeading`/`curvatureSpeedFactor`), and `track.ts` (`generateTrack`/`pointAt`/`nearestPoint`/`wallDistancesAt`) — anything that's actually a rule (math, state transitions) lives in a plain, Phaser-free module with its own unit tests; the `*Car`/`Projectile`/`Pickup`/`Hazard` classes are thin wrappers that own a sprite/physics body and call into that logic. `Projectile`, `Pickup`, and `Hazard` don't have an equivalent pure-logic split since they don't have any non-trivial rules of their own (they're driven entirely by config + the systems above).

`src/systems/` holds the cross-cutting orchestration `GameScene` used to do inline: each class owns one concern (lap/position bookkeeping, HUD drawing, keyboard input, hit-damage resolution, pickups) and is constructed once in `GameScene.create()`, fed per-frame state by `GameScene.update()`. This split happened once `GameScene` itself grew large enough that scrolling past unrelated concerns (e.g. HUD-drawing detail) to find the actual race logic became the bigger cost — the same "extract once a second instance of the problem shows up" reasoning the entity architecture section below already used for archetypes/weapons.

## Runtime architecture

- One `Phaser.Game` instance (`main.ts`), landscape 960×600 canvas (`CANVAS_WIDTH`/`CANVAS_HEIGHT` in `config.ts` — widened from the old portrait 480×720 now that the track winds sideways, not just up the screen, and a minimap will eventually want a free corner), Arcade Physics enabled, scene list `[BootScene, IntroScene, GameScene]`. `scale: { mode: Phaser.Scale.FIT, autoCenter: Phaser.Scale.CENTER_BOTH }` letterboxes that fixed logical resolution into whatever viewport it's given (e.g. a taller/narrower phone screen) without distorting gameplay; `input: { activePointers: 2 }` enables genuine two-finger multi-touch (default is 1 — a second simultaneous touch is otherwise silently dropped), needed for the virtual joystick plus a separate turret-aim/weapon-switch tap at the same time (see Touch controls below).
- **BootScene**: `preload()` loads cars/fx/hazards/audio from `public/assets/` and procedurally draws only the hazard-patch textures (rough/oil — see Rendering & assets). The road/track itself is no longer a pre-baked fixed-size texture; it's a different shape every race (see Track generation below), so `GameScene` draws it directly into a `Graphics` object at race start instead.
- **IntroScene**: a static controls/orientation screen (key/action pairs as plain text, no game state) that starts `"game"` on `keydown-SPACE` or `pointerdown`, the same dual-path restart pattern `GameScene`'s game-over overlay uses. Only reached once per page load — `GameScene.endGame()`'s restart calls `this.scene.restart()` directly on `GameScene`, bypassing `IntroScene` entirely, since a player who just died already knows the controls.
- **GameScene**: orchestrates —
  - one `generateTrack()` closed-loop track per race, drawn once into a static `Graphics` (see Track generation below) — the player spawns at the track's start point, oriented along its initial heading,
  - a `PlayerCar` driven by continuous WASD/arrow input (accelerate/brake-then-reverse/steer-by-heading), Shift to drift, 1/2/3 to switch weapons, Space/click to fire, or by the equivalent touch controls on a touch device (virtual joystick/fire button/tap-to-aim — see Touch controls below) — keyboard polling and weapon-select bindings live in `InputHandler`, touch controls in `TouchControls` (see Input below); the camera (`cameras.main`) follows the player around the loop, bounded to the track's bounding box, zoomed out a little (further on mobile) and biased each frame toward wherever the player is currently heading at speed (see Camera look-ahead below),
  - exactly `RIVALS.count` (5) `EnemyCar` instances, all real positioned cars active for the entire race (no on/off-screen activation, no timed spawn pool — see Track & rivals below), each running its own track-following + chase/avoid AI (see enemyBehaviors.ts),
  - player/enemy projectile, hazard, and pickup pools, plus `physics.add.collider`/`overlap` pairs for player↔enemies, player↔enemy-projectiles, player-projectiles↔enemies, enemies↔enemies, and player↔pickups; hazard contact is checked separately via an explicit per-frame `physics.overlap()` call rather than a registered pair (see Terrain hazards and obstacles below for why),
  - lap/race-progress tracking for the player and every rival, delegated to `RaceTracker` (see Track & rivals below), driving both the rubber-band system and the finish condition,
  - a text HUD (health, distance, best distance, speed, weapon+ammo, and a per-weapon aim/cooldown line — all called out in gameplay.md as a placeholder pending a later phase) plus graphical overlays (per-weapon aim meter ahead of the player along its heading, health bars below the player and above every active rival, the bottom-right weapon sidebar, and a temporary top-right lap/position debug readout), all owned and updated every frame by `HudSystem` (see Weapon visuals & in-world meter, Health bars and high score, Weapon sidebar),
  - run end → overlay → restart (space/tap or scene restart) via either `endGame()` (player health reaches zero) or `finishRace()` (player completes `TRACK.lapsToWin` laps), both funneling into the shared `showEndOverlay(title)`, which reads/writes a `localStorage`-persisted best distance (see Health bars and high score).

## Entity architecture: composition over deep inheritance

Built out once a second archetype actually needed different behavior, as planned:

- `EnemyArchetypeConfig` (in `config.ts`) holds each archetype's stats (health, speed multiplier, fire stats, score value) plus its own steering authority (`maxTurnRateDeg`, deg/sec — the 2D equivalent of the player's own turn rate) and `avoidanceWeight` (how hard it steers away from a close rival), plus behavior flags (`chasesPlayer`, `keepsDistance`) and a spawn `weight` (used once per rival slot when the 5-car roster is assembled at race start — see Track & rivals) — adding a 5th archetype is still a new config entry, not a new class. `lateralSpeed` (the old lane-drift quantity) is gone; Heavy's "barely dodges" identity is now expressed via a low `maxTurnRateDeg` and a near-zero `avoidanceWeight` rather than a literal zero — a literal zero would crash it into the first bend on a real 2D loop, unlike the old straight road. Contact sound/fx is still derived purely from whether the contact actually killed the enemy (explosion) or not (plain collision), the same rule `handleEnemyHitByProjectile` already used for weapon kills.
- `enemyBehaviors.computeTargetHeading()` is the swappable movement strategy, rewritten for the 2D loop: a pure function that starts from a *track-following baseline* (the heading toward a point `ENEMY_AI.lookaheadDistPx` ahead on the centerline — every archetype's job now, where the old straight-road model needed none), then blends in a pull toward the player's actual position (`chasesPlayer`, weighted down by distance beyond `ENEMY_AI.chaseRange`) or a push away from the player when too close (`keepsDistance`, within `ENEMY_AI.shooterPreferredGapPx`), then finally blends in a push away from whichever other rival is currently closest (within `ENEMY_AI.avoidanceRadius`), weighted by the archetype's own `avoidanceWeight`. When `atWall` is set (the car is pinned against a canyon wall — see Canyon walls above), all of that is skipped and the function returns the track-following heading outright: chase/avoid pulls can easily point straight back into the wall the car is already stuck against (the player parked just inside it, or another rival blocking the way off), so getting back on the road overrides them entirely rather than blending with them — without this a stuck car has no way back, or chases a moving target it can never reach and visibly spins in place. `enemyBehaviors.turnTowardHeading()` then rate-limits the car's actual heading toward that target by at most `archetype.maxTurnRateDeg * dt` — the same turn-rate-cap idea `playerPhysics` uses, replacing the old exponential-smoothing `smoothLateralVelocity`. `enemyBehaviors.curvatureSpeedFactor()` separately scales speed down for a sharp upcoming bend (the heading difference between the car's current heading and its own track-lookahead target), so AI doesn't take corners faster than its own turn rate can follow. All three are Phaser/track-free (they take precomputed headings/distances, not points or a `Track`) and unit-tested directly — `GameScene.updateRivals()` does the actual `Track` queries (lookahead point, nearest other rival) and passes the resulting angles into `EnemyCar.drive()`, which calls all three enemyBehaviors functions and resolves the result into `(vx, vy)` plus a rotation, the same way `PlayerCar.drive()` resolves heading+speed into velocity.
- Enemies also physically collide with each other via a `physics.add.collider(enemies, enemies)` pair (`GameScene.create()`) — the avoidance steering above reduces how often this triggers but doesn't prevent it, especially for aggressive archetypes that barely try to avoid in the first place. The player↔enemy pairing is a `physics.add.collider` too, for the same reason — see Collision shunt below for why that one specifically used to be an `overlap` and had to change.
- `weapons.WeaponController` plays the equivalent role for the player's weapons: ammo, cooldown, the side-gun sweep meter, and turret instability are all pure state transitions in one class, parameterized by `WeaponConfig`/`SIDE_GUN_SWEEP`/`TURRET_STABILITY` in `config.ts`. `PlayerCar` holds one instance and forwards handling state (steering/off-road/drifting) into it each frame; it doesn't know how any individual weapon works.
- This avoided a combinatorial subclass tree for either enemies or weapons — four enemy archetypes and three weapons exist as data + two small behavior modules, not seven classes.

## Systems (extracted from GameScene)

`GameScene` originally drove every cross-cutting concern itself directly. As more of those concerns landed (rivals' lap/position bookkeeping, the HUD's growing list of readouts/meters, keyboard input, hit-damage math, pickups), each one moved out into its own `src/systems/` class, constructed once in `create()` and fed per-frame state from `update()` — the same "don't build the abstraction until a second instance of the problem exists" reasoning the entity architecture above used for archetypes/weapons, applied to scene-level orchestration instead of game objects:

- **`RaceTracker`** owns `RivalState[]` (each rival's `EnemyCar` plus its lap-progress fields) and the player's own `playerStartS`/`playerLastS`. `initializePlayer`/`addRival` capture each car's unhinted spawn-time `s` as its lap-counting baseline; `updatePlayerS`/`updateRival` take each car's already-hinted-by-`nearestPoint` `s` for the frame (computed by `GameScene`, which owns the `Track`) and update lap/rubber-band state; `playerLaps`/`hasPlayerFinished`/`computeRacePosition` answer the questions `GameScene.update()` needs each frame to drive the finish condition and the debug position readout. See Track & rivals below for the lap-counting/rubber-banding rules themselves, which moved here unchanged.
- **`HudSystem`** owns every HUD `Text`/`Graphics` object (health/distance/speed/weapon/aim/race-debug text, the weapon meter, health bars, the weapon-sidebar highlight/reload-bar graphics and per-weapon ammo texts) and every per-frame draw call (`updateText`, `drawWeaponMeter`, `drawHealthBars`, `updateWeaponSidebar`, `updateRaceDebugText`). `GameScene` just calls these once per frame with the current `PlayerCar`/rival list/distance/speed; all the drawing detail (side-gun dual-mount arcs, turret crosshair, sidebar highlight) lives here. See Weapon visuals & in-world meter, Health bars and high score, and Weapon sidebar below.
- **`InputHandler`** registers and polls the keyboard keys (`W`/`S`/`A`/`D` + arrows + Shift + Space) and owns the `keydown-ONE/TWO/THREE` weapon-select bindings, returning a `PlayerInput` each frame from `getPlayerInput()` (track/hazard context is passed in fresh by `GameScene`, not tracked here — see Terrain hazards and obstacles for why that has to stay GameScene's call).
- **`CollisionHandler`** holds the pure damage/physics-resolution math shared by every way an enemy can take a hit: `damageEnemy(enemy, amount)` (apply damage, then the post-hit speed-slow debuff if it survived) and `resolveRamCollision(player, enemy)` (the closing-speed-to-shunt-magnitude math — see Collision shunt and ram damage below, which now lives here). `GameScene` still owns the *consequences* of a kill (sfx, explosion, despawn, scoring) since those are scene-level concerns this class has no business depending on.
- **`PickupSystem`** owns the pickup pool end-to-end: the chance-gated drop on enemy kill (`spawnOnEnemyDestroy`), the independent timed standalone-crate spawn (`updateCrateSpawn`, called from `update(delta)` — see Pickups below), applying a collected pickup's effect (`handlePickupCollection`), and the score-multiplier window that outlives the pickup object itself (`getScoreMultiplier`). Constructed with the scene's `Track` and seeded `rng` so crate placement and every roll in this class are part of the same seeded-race reproducibility `GameScene.getSeedFromUrl()` sets up (see Testing below).
- **`TouchControls`** owns the virtual joystick and fire button (mobile-only, gated by `isMobileMode()` — see `utils/device.ts`), plus two platform-agnostic bits that run on both touch and mouse: weapon-sidebar tap-to-switch (`onWeaponSelect` callback, same one `InputHandler` uses for the `1`/`2`/`3` keys) and `getTurretAimPointer()` (see Touch controls below). `GameScene.handleFiring()`/`HudSystem.drawWeaponMeter()` now resolve the turret's aim/fire pointer through this rather than reading `scene.input.activePointer` directly.

## Pickups

Implemented: `PickupSystem` (`src/systems/PickupSystem.ts`) plus the pooled `Pickup` entity. Two independent spawn paths feed the same pool:

- **Kill-drop**: `GameScene.destroyEnemy()` calls `pickupSystem.spawnOnEnemyDestroy(x, y)` on every enemy kill (weapon or ram), which rolls `PICKUPS.dropChanceOnKill` and then picks uniformly among all four `PickupType`s (`"health" | "ammo" | "boost-score" | "boost-speed"`).
- **Standalone timed crate**: `PickupSystem.update(delta)` ticks a `crateTimer` seeded from `nextCrateIntervalMs()` (`config.ts` — `CRATE_SPAWN.intervalMs` jittered by ± `CRATE_SPAWN.intervalJitterMs`); on expiry it re-arms by *adding* the next interval to whatever's left of `crateTimer` (rather than resetting outright, so an unusually large `delta` can't shorten the next interval), rolls a uniformly-random point on the track (`rng() * track.totalLength`, plus a random lateral offset margined in from the rock walls the same way `GameScene.spawnHazards()` margins hazard placements), and spawns a pickup there. Crates only ever roll `"health"` or `"ammo"` — boosts stay a kill-drop-only flavor bonus, per gameplay.md.

Collection (`PickupSystem.handlePickupCollection`, wired to a `physics.add.overlap(player, pickupSystem.getGroup(), ...)` pair in `GameScene.create()`) applies the effect and despawns the pickup: `health` calls `player.heal()`; `ammo` refills one weapon's ammo, picked at random per drop (`PICKUPS.ammoRestore[weaponType]`) — there's no per-weapon pickup, and no single pickup tops up the whole loadout; `boost-score` starts a `ScoreMultiplierEffect` (an end-timestamp + multiplier, checked against `scene.time.now` in `update()` rather than a `setTimeout`, consistent with everything else here being driven off the scene's own clock) that `GameScene.distanceTraveled` reads via `getScoreMultiplier()`; `boost-speed` calls `player.applySpeedBoost()`, which resolves to the same `maxForwardSpeedOverride` mechanism `DAMAGE_SLOW` uses (see Speed parity and damage-slow above — damage-slow wins if both are active).

## Pooling

All spawned-and-despawned object types are pooled Arcade Physics groups with a `maxSize` (enemies, player projectiles, enemy projectiles, pickups, hazards) — `group.get()` reuses an inactive instance (or creates one up to `maxSize`), and each entity has a `spawn()`/`despawn()` pair that resets its state instead of `destroy()`/recreate. This applies from the start for all of these types, per the "projectiles should be pooled from the start" and "pool once a second archetype with non-trivial reset state ships" guidance recorded here previously — both conditions are now true at once (weapons + three more archetypes landed together), so there was no reason to stage it.

One-off visual effects (the explosion sprite on a kill) are *not* pooled — they're rare relative to projectiles/enemies, created with a tween that self-destroys, and not worth the bookkeeping.

## Collision shunt and ram damage (ramming no longer costs the player health, but now costs the enemy)

`CollisionHandler.resolveRamCollision()` (`src/systems/CollisionHandler.ts` — called from `GameScene.handlePlayerEnemyCollision()`, which then just handles the kill's sfx/explosion/despawn) converts the *actual relative closing speed* of the impact into a magnitude — in the 2D model that's the relative velocity between the two cars projected onto the line connecting them (`-(relVx·dx + relVy·dy) / dist`, where `dx, dy` is the enemy-minus-player position vector), the 2D generalization of the old straight-road version's `Math.abs(enemy.body.velocity.y)`. `COLLISION_SHUNT` (`config.ts`) floors that at `COLLISION_SHUNT.minShunt` so even a near-zero-relative-speed bump still registers as something. Ramming still only knocks the player's *speed* around, never costs health directly. `speedFactor`/`minShunt` were raised and `ramDamageFactor` lowered from their original values — playtesting feedback was that a ram read more like a number changing than a physical hit, so the impact magnitude (and the speed jolt it drives) is bigger now, traded against a bit less of the enemy damage that same magnitude also drives (see below).

That magnitude is applied via `PlayerCar.applyImpactShunt(signedAmount)`, which adds a signed delta directly to `forwardSpeed` — it would do nothing useful to dampen-toward-zero logic for a player sitting still (a stationary player rammed by a fast chaser should still visibly lurch). Direction is now derived from where the enemy sits relative to the *player's own heading* (`forwardDot = dx·sin(headingRad) - dy·cos(headingRad)`, the dot product of the enemy's relative position against the player's forward unit vector) rather than a world-axis comparison — an enemy behind the player's direction of travel (`forwardDot < 0`) jolts them forward, one ahead of it jolts them back. This is the direct 2D generalization of the old `enemy.y >= player.y` check, which only worked because the player's heading was always "up." The next frame's normal drive-speed clamp folds the result back into the valid range, so no separate clamping is needed here.

The *same* magnitude, scaled by `COLLISION_SHUNT.ramDamageFactor`, is also applied to the enemy via `CollisionHandler.damageEnemy()` (`takeDamage()` then, if it survived, `applyDamageSlow()` — see Speed parity and damage-slow below) — ramming is just another damage source feeding the same health pool as weapon fire, not a separate instant-kill rule. The enemy is only destroyed (explosion fx/sfx, despawn — `destroyEnemy()`, same path a weapon kill uses) if that brings its health to zero; otherwise it just plays the plain collision sound and keeps driving, now hurt and temporarily slower. `EnemyCar.ramCooldown` (set to `COLLISION_SHUNT.ramCooldownMs` on every ram, decremented in `drive()`) guards `handlePlayerEnemyCollision()` against re-applying ram damage on the next frame or two of continued overlap. Enemy gunfire and canyon-wall impacts are unaffected by any of this and still cost health normally; off-road driving never costs health at all (see Canyon walls above).

**Player↔enemy contact is a `physics.add.collider`, not an `overlap`** — Arcade Physics needs to physically push the two bodies apart every step they touch (on top of invoking the ram-damage callback), since a surviving enemy would otherwise just keep moving at its own AI-driven velocity, unaffected by the "collision," and glide straight through the player with nothing to stop it (an `overlap` only reports contact, it never separates bodies).

**The collider alone isn't enough to make the bounce-apart feel reliable, though** — a chasing archetype's AI doesn't know it just rammed, and left alone would simply recompute the same closing target next frame and get driven straight back into contact, fighting the collider every step. The fix is in `EnemyCar.drive()`: while `ramCooldown > 0`, it skips its normal AI entirely and instead moves directly away from the player along the real player→enemy vector (`(dx/dist, dy/dist) * COLLISION_SHUNT.recoilSpeed`) and turns to face that direction, for the same `ramCooldownMs` window the damage-reapplication guard above uses. Unlike the old straight-road version (which could only recoil along a fixed axis, "behind" cars falling back in y and "side" cars peeling off in x), this works for a ram from any angle, which the 2D loop needs.

## Speed parity and damage-slow

Enemy speed is `RIVALS.baseApproachSpeed` × the archetype's own `speedMultiplier` × several situational multipliers (rubber-band, wall contact, damage-slow, and now also curvature — see `enemyBehaviors.curvatureSpeedFactor`) — there's no more "approach speed relative to the player's forward speed" concept; with the world no longer scrolling relative to a fixed player position, each car (player and rivals alike) just drives in real 2D space at its own speed, and the relative motion everyone actually sees falls out of that on its own. `RIVALS.baseApproachSpeed` (600) × the aggressive archetypes' `speedMultiplier` (chaser 1.05, bomber 1.1) lands close to — bomber just past — the player's own `maxForwardSpeed` (620), so neither side has a structural speed advantage; heavy and shooter stay deliberately slower as part of their archetype identity (see Enemies in gameplay.md), not as a general rule. Raised from an initial 560 after playtesting found rivals read as sluggish/passive even before factoring in archetype multipliers — the same playtesting pass also raised every archetype's `maxTurnRateDeg` and `ENEMY_AI.chaseWeight`/`chaseRange`, so chasers commit to a ram attempt more decisively instead of mostly just track-following near the player.

With speed roughly symmetric, the thing that actually decides a chase needed to be combat, not the spawn config — that's `DAMAGE_SLOW` (`config.ts`). Any discrete hit that *doesn't* kill its target — a weapon hit or ram-damage hit landing on an enemy that survives, or an enemy projectile hitting the player — temporarily caps the victim's top speed (`maxSpeedFactor`) for `durationMs`, on top of an immediate one-time knock to the player's *current* speed specifically (`speedMultiplier`, applied once in `PlayerCar.applyDamageSlow()` since the player has a persistent `forwardSpeed` value to multiply; enemies don't — their "speed" is recomputed fresh from config every frame in `EnemyCar.drive()`, so the timer-driven cap alone is the whole effect there). On the enemy side, `applyDamageSlow()` is called from exactly one place — `CollisionHandler.damageEnemy()`'s non-lethal branch — shared by both `handleEnemyHitByProjectile()` (weapon hits) and `handlePlayerEnemyCollision()`'s ram-damage path (`resolveRamCollision()` also funnels through `damageEnemy()`), so the two damage sources can't drift out of sync. On the player side, `GameScene.damagePlayer()` (enemy gunfire hitting the player) calls `PlayerCar.applyDamageSlow()` directly — distinct from the wall-impact damage path, which calls `player.takeDamage()` directly from `PlayerCar.drive()` and bypasses `damagePlayer()` entirely (off-road driving itself never calls `takeDamage()` at all — only the drag in `computeDrive()`, no health cost).

`PlayerCar`'s `damageSlowTimer` and `speedBoostTimer` (from the speed-boost pickup) both resolve to a `maxForwardSpeedOverride` passed into `computeDrive()` — if both are active at once, the damage slow wins, since getting hit should always feel impactful regardless of buffs. `EnemyCar.drive()` decrements `damageSlowTimer` unconditionally at the top, before the ram-cooldown recoil branch's early return, so it keeps counting down even during a recoil window rather than effectively pausing (which would make the visible debuff outlast `durationMs` by however long the recoil happened to run).

## Game modes: Arcade vs. Repair, and the mid-race restart button

`GameMode` (`config.ts`, `"arcade" | "repair"`) is chosen once on `IntroScene` and carried through every later `scene.restart()` this session via `GameScene.init(data)`, which falls back to the existing `this.mode` field when `data` omits it — the Scene instance (and its plain fields) survives a `restart()` the same way `gameOver`/`won` etc. already needed manual resets in `create()` for, so the mode persists across restarts without IntroScene's own involvement. `IntroScene` itself always re-defaults to `"arcade"` on a fresh visit (i.e. a real page load) rather than remembering a previous session's choice — switching modes deliberately requires going back through IntroScene, not just restarting.

The mode is consumed entirely inside `PlayerCar`/`EnemyCar.takeDamage()`: both clamp health to 0 and, in repair mode, set an `isRepairing` flag + start a `REPAIR_MODE.repairDurationMs` countdown *instead of* returning `true` (destroyed). Every existing call site — `CollisionHandler.damageEnemy`/`resolveRamCollision`, the wall-impact damage in `drive()`, `GameScene.damagePlayer()` — already just branches on that boolean to decide whether to run kill effects (explosion/`endGame()`), so none of them needed to learn about modes directly; arcade mode's behavior is completely unchanged, repair mode just makes `takeDamage()` return `false` for one extra reason. `drive()` then short-circuits at the very top while `isRepairing` (zero velocity, no input/AI, no firing — `EnemyCar.canFire()` also checks it explicitly, since a stale `fireTimer`/`distanceToPlayer` from before repairing started could otherwise pass), decrementing the same timer and restoring `REPAIR_MODE.healthRestoreFraction` of max health (clearing the gray `0x888888` tint back to the archetype's own tint, or `clearTint()` for the player) once it elapses. `GameScene.update()`'s own end-of-run check (`player.health <= 0`) is gated on `!player.isRepairing` so a player mid-repair doesn't trigger `endGame()` the instant health hits 0 in the same frame `takeDamage()` already started the repair.

The mid-race **RESTART** button (HUD top-center, also bound to the `R` key via `InputHandler.isRestartJustPressed()`, which uses `Phaser.Input.Keyboard.JustDown` rather than `isDown` specifically so holding `R` through the restart can't immediately re-trigger another one on the new scene's first frame) is gated on `GameScene.stationaryTimer` — accumulated whenever `|forwardSpeed| <= RESTART_BUTTON.stationarySpeedThreshold`, reset otherwise — crossing `RESTART_BUTTON.stationaryDurationMs`. It's independent of `GameMode`; both arcade and repair races can use it, restarting via `scene.restart({ mode: this.mode })` the same way the game-over/finish overlay's restart bindings do, so a mid-race restart (or a death/finish restart) never silently falls back to arcade.

## Terrain hazards and obstacles (rough road, oil slicks, small rocks)

Hazards are placed once around the loop at race start (`GameScene.spawnHazards()`), not spawned continuously down a scrolling road the way the old endless-runner model did — the track is fixed for the whole race now, so there's nothing to continuously generate. `HAZARDS.roughCount`/`oilCount` + `OBSTACLES.count` placements are shuffled into a list, then walked around the track at roughly even arc-length slots (skipping `HAZARDS.startClearancePx` right at the start/finish line) with random jitter within each slot and a random lateral offset within the paved width (margined in from the rock walls by `HAZARDS.lateralMargin`) — even spacing without a perfectly mechanical grid. All placement randomness goes through `this.rng` (see Testing below for why).

`HazardType` is `"rough" | "oil" | "obstacle"`. "rough"/"oil" are unchanged from the original design — continuous patches a car drives over, still affecting whoever's overlapping every frame, never despawning on contact. "obstacle" is new: a small discrete rock/debris pile (`OBSTACLES` in config.ts) that's a one-time bump — flat damage plus a one-time speed penalty — and despawns itself on first contact, since it's a discrete thing you clip, not ground you linger on (closer to the old pre-rough-terrain-rework debris/barrier model, reintroduced as a third hazard type alongside the continuous patches rather than replacing them). Textures are irregular polygon blobs (`BootScene.drawIrregularBlob()`, sized entirely off `OBSTACLES.size` — no other hardcoded pixel dimension or physics-body size to keep in sync, the Arcade body auto-derives from the generated texture); the obstacle texture reuses the canyon wall's rock palette (`WALLS.rockColor`/`rockSpeckleColor`) at a much smaller size, rather than a scaled-down rough-patch texture, so it reads as a distinct rock, not a tiny patch. `size` was bumped up and `damage`/`speedPenaltyFactor` retuned (more one-time speed cut, a bit less flat damage) after playtesting found the original rock too small/inconsequential to register as a real obstacle — the same speed-over-damage trade as the ram tuning above, and as the new one-time wall-impact speed cut (`WALLS.impactSpeedPenaltyFactor`, applied in `PlayerCar.drive()`/`EnemyCar.drive()` on the same rising edge as the existing one-time wall damage, in addition to — not instead of — the wall's pre-existing continuous scraping drag).

Hazard contact is checked via an *immediate* `this.physics.overlap(...)` call inside `update()`/`updateRivals()` rather than a `physics.add.collider`/`overlap` pair registered once in `create()`. This matters specifically here (and nowhere else in the scene) because the result feeds back into the *same frame's* `PlayerInput` before `player.drive()` runs — a registered pair's callback timing relative to our own `update()` isn't something we want this effect's correctness to depend on, since calling `physics.overlap()` ourselves at a known point removes that dependency entirely instead of reasoning about it.

- **Player** (`handlePlayerHazardOverlap`): `this.onRoughTerrain` is reset to `false`, then the explicit overlap check sets it back to `true` if still on a rough patch, all before building that frame's `PlayerInput` — so `computeDrive()` (`playerPhysics.ts`) sees a value from *this* frame, not a frame-old one. It applies extra drag (`ROUGH_TERRAIN.dragPerSecond`) whenever true, the same `decayTowardZero` mechanism off-road drag already uses. `this.oilSlickTimer` is refreshed to `OIL_SLICK.effectDurationMs` by the same check every frame still overlapping an oil patch, then counted down once the player leaves it, so the effect lingers briefly after exit. `computeDrive()` reduces turn authority (`OIL_SLICK.controlMultiplier`) and biases velocity direction by `input.oilDriftBias * OIL_SLICK.driftStrengthDeg` whenever `oilSlicked` is true — turns less responsively *and* slides off-heading, "harder to control," not just "turns slower." `oilDriftBias` (a value in `[-1, 1]`) is rolled once when `oilSlickTimer` transitions from inactive to active, not every frame — holding one rolled value for the whole activation makes the car visibly drift/slide in one direction for the duration, the way losing traction actually feels (an earlier version re-rolled every frame and just read as a vibration in place). Obstacle contact is a flat `damagePlayer(OBSTACLES.damage)` (the same path enemy gunfire uses, including the temporary top-speed cap) plus `PlayerCar.applyObstacleHit(OBSTACLES.speedPenaltyFactor)` — a one-time multiply on current speed, distinct from the damage-slow timer, since clipping a rock is a single physical bump, not a lingering debuff.
- **Rivals** (`handleRivalHazardOverlap`): `GameScene.updateRivals()` resets every active rival's `onRoughTerrainThisFrame` to `false`, then runs one `physics.overlap(enemies, hazards, ...)` call per frame covering all active rivals at once. `EnemyCar.drive()` applies the matching effects directly — rough terrain scales down that frame's `ownSpeed` by `ROUGH_TERRAIN.enemySpeedMultiplier`; an active oil-slick timer reduces that frame's turn rate by `OIL_SLICK.controlMultiplier` (same as the player) and adds `oilDriftBias * OIL_SLICK.enemyDriftStrengthDeg` directly onto the computed heading, on top of (not folded into) the normal rate-limited turn, so it reads as a sudden loss of control rather than something the turn-rate cap would otherwise ease away. Obstacle contact is `takeDamage` + `applyDamageSlow()` (no separate speed-penalty multiply — enemies don't have a persistent `forwardSpeed` to multiply the way the player does; the existing damage-slow timer already covers "temporarily slower after a hit" for them, see Speed parity and damage-slow above).

Neither rough nor oil costs health for player or enemies; obstacles do (a flat one-time amount, not a per-second drain) — that's the enemy-fire/wall/obstacle-vs-patch split (off-road driving itself is drag-only, never a health cost — see Canyon walls above).

## Track generation (closed-loop canyon)

`src/entities/track.ts`'s `generateTrack(rng?)` (unit-tested in `track.test.ts`, an injectable `rng` defaulting to `Math.random` so tests can be deterministic) replaces the old infinite-meander model with an actual closed loop:

1. **Control points**: `TRACK_GEN.controlPoints` points placed around a circle (`TRACK_GEN.baseRadius`), each with a randomized radius (`radiusJitter`) and a randomized angle nudge capped to a fraction of its own angular segment (`angleJitterFraction`) — capping the angle nudge keeps points strictly ordered around the circle, which is what guarantees the closed spline through them never self-intersects.
2. **Dense spline sampling**: a closed Catmull-Rom spline through the control points (manual pure-JS implementation, not `Phaser.Curves`, to keep this module Phaser-free per the project's pure-logic-module convention), densely sampled (`TRACK_GEN.segmentsPerControlPoint` steps per control-point segment) — this dense sampling is an intermediate, not the final centerline, since the spline's own parameter isn't evenly spaced by actual distance.
3. **Arc-length resampling**: the dense samples are walked once to build a cumulative chord-length table, then resampled at fixed `TRACK_GEN.sampleSpacing` px intervals into the final centerline (`Track.samples: TrackSample[]`, each `{x, y, headingRad, s}`) — every consumer downstream (wall rendering, AI lookahead, lap progress) can treat `s` as a uniform physical distance along the loop rather than an uneven spline parameter. Heading at each sample faces the next one (wrapping for the last sample back to the first).

`Track` also carries `pavedHalfWidth` (constant along the loop, from `ROAD_WIDTH / 2`) and a `bounds` box (sample extents padded by the max possible wall offset) used for the camera's `setBounds`.

Two query functions consume a `Track`:

- **`pointAt(track, s)`** — wrapping interpolation along the centerline at an arbitrary arc-length position (handles `s` outside `[0, totalLength)` by wrapping). Used for the player's/rivals' spawn points and AI lookahead targets.
- **`nearestPoint(track, x, y, hintS?)`** — finds the closest point on the centerline *polyline* to an arbitrary world position via a linear scan over samples (a few hundred for a typical track — fast enough for a handful of cars per frame without spatial indexing) refined against the two segments adjacent to the nearest sample. Returns `{s, lateralOffset, distance}`, where `lateralOffset` is signed (positive = right of the direction of travel at that point, via the dot product against the heading's right-hand normal `(cos(headingRad), sin(headingRad))`). This is the per-frame replacement for the old `x < ROAD_X || x > ROAD_X + ROAD_WIDTH` off-road check — `GameScene.update()`/`updateRivals()` call it once per frame on each car's position and feed the result into `PlayerInput`/`EnemyDriveParams`. See Track & rivals below for what the optional `hintS` is for and why it matters.

**Canyon walls**: `wallDistancesAt(track, s)` reuses the old sine-meander formula (`WALLS.meanderFreqLeft/Right`, `WALLS.rightPhaseOffset`, clamped to `[WALLS.minOffset, WALLS.maxOffset]`) unchanged in spirit — only the domain changed, from world-scrolled-distance to arc-length `s`. It returns each side's distance *from the centerline* (`pavedHalfWidth + meander offset`), which `playerPhysics.computeDrive()` compares against `lateralOffset` to derive `isOffRoad`/`atWall` and apply the continuous extra drag for either — there's no more lateral-velocity-zeroing inside `computeDrive` for this; see Heading-based steering below for why that moved to a position clamp in `GameScene` instead. Health damage is handled separately by the caller (`PlayerCar.drive`/`EnemyCar.drive`): off-road never costs health, only the continuous extra drag above — a wall hit is the *only* terrain damage, a one-time `wallImpactDamage()` (in `config.ts`) applied only on the frame contact begins, scaled by speed at that instant, so lingering against a wall while steering free doesn't keep draining health the way the drag does.

**Rendering**: since the whole loop is fixed for the entire race (not an infinite scroll), `GameScene.drawTrack()` draws it **once** at race start into a single static `Graphics` (depth `DEPTHS.roadBackground`) — a filled quad per consecutive sample pair for the paved road surface, a quad on each side out to that sample's wall distance for the rock band, and a periodic dashed quad for the centerline divider (`DIVIDER_DASH_PERIOD`/`DIVIDER_DASH_FRACTION`) — strictly simpler than the old model's per-frame redraw (`drawWalls()`) plus scrolling `TileSprite` divider, neither of which exists any more. `BootScene` no longer pre-bakes a `road-background`/`road-divider` texture at a fixed canvas size, since the track's shape differs every race.

## Track & rivals (always-active 2D rivals, lap counting, the start/finish seam)

The old `RivalSlot`/`gapPx`/on-off-screen activation model (a 1D screen-relative gap, since that's all a fixed vertical lane needed) is gone. `GameScene.spawnRivals()` builds exactly `RIVALS.count` (5) `EnemyCar`s once, in `create()`, positioned via `RIVALS.startOffsets` (`{s, lateral}` pairs placing them in a small grid just behind the player's own start point) and hands each one to `raceTracker.addRival()` — every rival is a real positioned 2D car, active for the whole race, the same way the player is. `RivalState` (`src/systems/RaceTracker.ts`) holds each one's `car`, lap-progress fields, and `rubberBandTimer`; `RaceTracker` is the one class that owns this bookkeeping for both the player and every rival (see Systems above).

**Lap counting and the start/finish seam.** `s=0` and `s=totalLength` are the same physical point on a closed loop, so `nearestPoint`'s nearest-segment search can legitimately resolve a car sitting at the start/finish line to *either* representation — and in practice it does flip between them from one frame to the next on even a sub-pixel move, since both candidate segments are within float noise of being equidistant right at the seam. An early version of lap counting (`hasCompletedLap(lastS, currentS, totalLength)`, detecting a lap as "`s` jumped backward by about half the track") broke on exactly this: a car sitting still at the start line could register a phantom completed lap from the representation flip alone, with zero real movement. The fix is `nearestPoint`'s optional `hintS` parameter: pass the previous frame's own `s` back in, and the function shifts its raw result by whichever multiple of `totalLength` keeps it within half a lap of the hint, so `s` drifts continuously near the seam instead of ever jumping. Each car tracks two values — `lastS` (continuously unwrapped via the hint, fed back in as next frame's hint) and `startS` (the *unhinted* value captured once at spawn) — and laps is always `Math.floor((lastS - startS) / totalLength)`, correct regardless of which seam representation `startS` happened to land on, since it's a delta from spawn, not an absolute position. Total race progress for ranking/rubber-banding purposes is just `lastS - startS` directly (no separate "laps × totalLength + s-within-lap" computation needed — the continuously-unwrapped `lastS` already *is* that value).

`GameScene.updateRivals()` runs each frame: queries `nearestPoint` (hinted), applies the same belt-and-suspenders wall clamp the player uses (`clampToTrackBounds`, shared between both), calls `raceTracker.updateRival()` to update rubber-band state and get back this frame's speed multiplier, computes a lookahead point and the nearest other rival (excluding the player — chasing archetypes are *supposed* to close on the player, so it shouldn't count as something to avoid), and calls `EnemyCar.drive()` with all of it. A rival that dies (weapon, ram, or wall damage) just goes inactive (`despawn()`) — there's no separate "alive" flag the way the old `RivalSlot.alive` needed, since Arcade's own `active` flag already means exactly that now that there's no off-screen deactivation to confuse it with.

**Rubber-banding is bidirectional** (`RivalState.rubberBandMultiplier`, `RUBBER_BAND` in config.ts, resolved in `RaceTracker.updateRival()`) — the original version only ever slowed down a rival that had pulled ahead, which meant a rival that fell behind (a slower archetype especially) had no way back into contention for the rest of the race. Each rival independently rolls `RUBBER_BAND.activationChancePerSecond * dt` while its `rubberBandTimer` is at 0; on activation, `lead = rivalProgress - playerProgress` decides which multiplier gets stored — `aheadSpeedMultiplier` (<1, slows it down) if `lead > minLeadPx`, `behindSpeedMultiplier` (>1, speeds it up) if `lead < -minDeficitPx`. The stored multiplier (not just a boolean timer) is necessary because the timer alone can't say which direction triggered it. Each rival rolls independently, so it reads as that car catching a break, not a global rubber band snapping every rival to the same gap.

`RaceTracker.computeRacePosition()` sorts currently-active cars (player + surviving rivals) by total progress to get a live finishing position — read by `GameScene` and shown via `HudSystem` as a temporary debug readout (`raceDebugText`) until a later phase's real HUD. The finish condition (`GameScene.update()`) is simply `raceTracker.hasPlayerFinished()` — the player's own lap count reaching `TRACK.lapsToWin`.

**Why Shooter's shots seemed to only land "when it was in front."** It wasn't a facing/position gate — `canFire()`/`spawnEnemyProjectile()` have never checked where the player is relative to the rival. The actual cause: Shooter is the only firing archetype, is slower than baseline (`speedMultiplier: 0.85`), and (before the bidirectional rubber-band fix above) had no way to close a gap once it fell behind — combined with `PROJECTILE_LIFETIME_MS` (2000ms) capping its `projectileSpeed` (480) shots at ~960px of travel, a Shooter that had fallen behind by more than that was firing into the void every cooldown, never able to land a hit regardless of aim. Close encounters — which a slower, falling-behind car mostly gets right after the grid start, while it's still near the player and roughly able to keep pace — were the only situation where shots could physically reach, which read as "only fires when in front." Both contributing factors are addressed above (rubber-banding keeps it close more consistently) and via a raised `PROJECTILE_LIFETIME_MS` (2800ms, ~1344px of range at Shooter's projectile speed).

## Heading-based steering

`playerPhysics.computeDrive()` was rewritten from a lane-drift model (steering set an x-velocity directly, the car sprite never rotated) to a heading model, matching the move from a fixed vertical lane to a free 2D closed loop:

- **State** (`PlayerDriveState`): `forwardSpeed` (signed scalar, accel/brake/reverse curves unchanged in spirit from the old model), `headingDeg` (canvas-angle convention, 0 = up/+clockwise — the same convention the weapon-aim math already used), and `velocityHeadingDeg` (the car's *actual* direction of travel, which can lag behind `headingDeg`).
- **Turning**: steering changes `headingDeg` at a speed-scaled rate (`PLAYER_HANDLING.minTurnRateDeg`/`maxTurnRateDeg`, deg/sec — the same min/max-ramp-by-speed-fraction idea the old lateral model used, just angular now), multiplied by `DRIFT.turnMultiplier` while drifting and `OIL_SLICK.controlMultiplier` while oil-slicked. Steering while reversing flips the turn direction (`reverseFlip = speed < 0 ? -1 : 1`), matching how backing up a real car works.
- **Drift slip**: `velocityHeadingDeg` eases toward `headingDeg` each frame rather than snapping to it — fast (`PLAYER_HANDLING.velocityGripEase`) while gripping, much slower (`DRIFT.slipEase`) while drifting or oil-slicked. This lag is what makes a drift read as an actual slide (the car's momentum keeps carrying it along the old heading for a beat) rather than just a sharper turn. Oil-slick additionally biases the *target* by `oilDriftBias * OIL_SLICK.driftStrengthDeg` (degrees now, not a raw px/sec push, since velocity direction is heading-relative).
- **Output**: `(vx, vy)` resolved from `velocityHeadingDeg` and `forwardSpeed` (`vx = sin(rad) * speed`, `vy = -cos(rad) * speed`), applied via `PlayerCar.drive()`'s `setVelocity(vx, vy)` — Arcade Physics still owns position integration, only the velocity source changed. `PlayerCar` also calls `setRotation(headingRad)` every frame; this works without any conversion because Phaser's own sprite-rotation convention (0 = the texture's unrotated/"up" orientation, positive = clockwise on screen) is the same canvas-angle convention used everywhere else in this codebase.

Wall/off-road position correction is no longer inside `computeDrive` (it has no track-normal vector to push along) — `GameScene.update()` does it instead: re-running `nearestPoint`/`wallDistancesAt` after `drive()` and, if `lateralOffset` exceeds the wall distance on either side, repositioning the car along the track-normal direction at that arc-length position. This is the direct equivalent of the old model's `Phaser.Math.Clamp(player.x, ...)` belt-and-suspenders guard, just expressed in track-relative coordinates instead of world x.

Rocket/side-gun firing math (`GameScene.spawnPlayerProjectile`) and the in-world weapon meter (`HudSystem.drawWeaponMeter`) both now resolve angles relative to `player.heading` rather than a fixed "up" — see Weapon visuals & in-world meter below.

## Rendering & assets

Mixed, deliberately: loaded CC0 assets for anything where sourced art is a clear win, procedural generation kept for the road/ground specifically to control palette consistency (see CLAUDE.md's Assets section for the policy this implements).

Render depth is centralized in `DEPTHS` (`config.ts`) and applied once per type in each entity's constructor (`EnemyCar`, `Hazard`, `Pickup`, `Projectile`, `PlayerCar`), rather than left at Phaser's default of 0 for everything. Without explicit values, same-depth objects tie-break by display-list insertion order, which depends on pool creation order — that's what let hazard patches render in front of cars depending on spawn timing. Ground decals (`hazard`, `pickup`) sit below `enemy`/`player`; `projectile` sits above both so a shot flying over a car right before impact stays visible.

- **Cars** (`public/assets/cars/`): FieraRyan's CC0 "Futuristic Wasteland Top-down Cars" — 5 of the pack's 9 designs, picked for visual variety, preprocessed once via `scripts/process-car-sprites.mjs` (rotated from the source's landscape orientation to portrait, flipped for archetypes that need to visually face the player, downscaled from ~1170px to 90px tall — shrunk from an initial 128px pass once cars on-screen read as too large/cluttered). `EnemyCar.spawn()` applies a per-archetype `tint` (in `ENEMY_ARCHETYPES`) since the source pack has no color variants of its own — the player car is left untinted. Both `PlayerCar` and `EnemyCar` additionally apply `CAR_SCALE` (0.6) in their constructors, on top of that preprocessing, to leave more visible road on the fixed-size canvas.
- **Fx** (`public/assets/fx/`): Kenney's Particle Pack (explosion/projectile/pickup glow shapes — these are grayscale by design, meant to be recolored) plus an ammo-pickup icon from Kenney's Racing Pack. `GameScene` applies `VISUAL_TINTS` (in `config.ts`) plus `Phaser.BlendModes.ADD` for the glow-style fx at each spawn site, both to recolor Kenney's brighter house palette toward this game's grimmer one and, for the particle shapes, to get the intended glow look. Player projectiles specifically use `WEAPON_VISUALS` (tint + scale per weapon) instead of the shared fx tints, so the three weapons stay visually distinct from each other (see Weapons below).
- **Road/ground**: procedural, but no longer a pre-baked `BootScene` texture — the track's shape is different every race (see Track generation above), so `GameScene.drawTrack()` draws it directly with `Graphics` fill calls at race start using the same `ROAD_COLORS` palette (dark cracked-asphalt road, dirty-yellow divider) plus `WALLS.rockColor` for the canyon walls; the camera's background color stands in for the old dead-ground texture. **Terrain hazards**: still `Graphics.generateTexture()` in `BootScene`, not sourced — Kenney's matching road tiles read as bright kart-racer colors that clash badly with the gritty cars, and a flat recolor of our own simple shapes was both consistent and cheap. `TERRAIN_COLORS` in `config.ts` holds the rough-road/oil-slick patch palette — change it there, not by hunting for new art. The discrete debris/barrier sprites from Kenney's Racing Pack used before the rough-terrain/oil-slick rework have been removed (no sourced art fits a ground-level "patch you drive over" shape).
- **Audio** (`public/assets/audio/`): Kenney's Impact Sounds / Sci-fi Sounds / Interface Sounds, loaded via `this.load.audio()` under the same `SFX_KEYS` the previous Web-Audio-synthesis version used, so `GameScene.playSfx()` didn't need to change at all.

Full source list and license notes: `CREDITS.md`. No continuous/looped engine sound; only discrete one-shot effects (collision, explosion, rocket, gunfire, pickup, game over).

### Weapon visuals & in-world meter

The three weapons previously differed only by a faint tint on the same shared bullet sprite, which made the side guns and turret hard to tell apart in motion. `WEAPON_VISUALS` (`config.ts`) now gives each weapon its own tint, projectile scale, and blend mode — rocket (large, saturated red-orange, `normal` blend, scale 2.4), side guns (pale yellow, `add`, scale 1.3), turret (small, cyan, `add`, scale 0.9) — applied in `GameScene.spawnPlayerProjectile()`. All three scales were bumped up from their original values after playtesting found bullets hard to spot once cars (and so the overall scene) shrank to `CAR_SCALE`.

The rocket specifically needed `normal` blend instead of `add`: it fires dead-ahead along whatever the car's current heading is (see Heading-based steering above — `fireAngleDeg` adds `player.heading` to the weapon's relative angle for rocket/side-guns), which crosses the dashed centerline divider often, and an additively-blended light-colored glow washes out to near-white — i.e. effectively invisible — against it. Since the rocket fires rarely (long cooldown, 6 shots) each shot disappearing like that reads as "the weapon doesn't work," unlike the rapid-fire side guns/turret where one faint frame is lost in the stream. `normal` blend with a saturated tint keeps the rocket visibly distinct from both the dark road and the pale divider.

`HudSystem.drawWeaponMeter()` draws a small `Graphics` overlay floating just ahead of the player car along its current heading each frame (anchor computed via `aimPointOnArc(player.x, player.y, anchorDist, 0)`, not a fixed screen-up offset, so it turns with the car), shape depending on the equipped weapon:

- **Rocket**: no in-world meter any more — dead-ahead-only firing has no aim state worth showing here. Its reload progress moved into the bottom-right weapon sidebar instead (see Weapon sidebar below), inline with its ammo count.
- **Side guns**: two mirrored dots, each sweeping its own arc centered outward at `±SIDE_GUN_MOUNTS.centerAngleDeg` (90°, relative to heading) rather than around 0°/forward, positioned at each side mount via `sideGunMountPosition(player.x, player.y, player.heading, player.displayWidth, side)` — **from the player's actual position**, not from the meter's own forward-offset anchor point `(cx, cy)` used elsewhere in `drawWeaponMeter`. That distinction matters: an earlier version of this code based the mount center on `(cx, cy)` (the point floating ahead of the car for the rocket/turret-style meter anchor), which added that forward offset on top of the sideways mount offset and visibly detached the arcs/markers from the actual barrels — caught in playtesting as "the sidegun overlay isn't aligned with the car." Side guns have no forward offset in real life (they mount at the car's sides), so their meter has to anchor at the car's position, unlike the rocket/turret meters which are genuinely supposed to float ahead. `sideGunMountPosition`/`sideGunAngleDeg` (both pure functions, `src/entities/weapons.ts`) are shared between this meter and `GameScene.spawnPlayerProjectile()`'s actual projectile spawn point — they used to be near-duplicated private methods on both `GameScene` and `HudSystem`, kept in sync by hand; consolidating them into one pair of functions is what guarantees the meter and the actual firing angle always agree, rather than just being expected to. Both markers derive from the one shared `sweepAngleDeg` via `sideGunAngleDeg(side, sweepAngleDeg)` — `-90 + sweep` for the left mount, `90 - sweep` for the right. This direction flip (outward instead of forward) followed directly from playtesting: mounting the barrels at the sides but still sweeping them around a forward-centered arc made the weapon *look* like it was still firing forward in a cluster, not reading as side-mounted at all. `SIDE_GUN_SWEEP.maxAngleDeg` was also narrowed (55° → 40°) after playtesting found the original arc swung too far back toward the rear at its extremes.
- **Turret**: the existing widening/narrowing spread cone (centered on heading, not the absolute aim direction — it's showing spread magnitude, not exact aim), plus a crosshair (`strokeCircle` + cross lines) drawn at `scene.input.activePointer`'s world position — added because the turret's aim direction otherwise had no visible indicator at all beyond the spread cone, which doesn't show *where* on screen the pointer actually is. Turret aim itself stays absolute (independent of heading — see `fireAngleDeg`), since it's aimed directly at the pointer's world position regardless of which way the car is facing.

This is a glanceable supplement to the existing numeric HUD text (`aimText`), not a replacement — the corner text stays the precise readout. The meter (`HudSystem`) and the firing-angle math (`GameScene.spawnPlayerProjectile`/`computeTurretAimDeg`) share one canvas-angle convention (0° = straight ahead/heading, positive = clockwise) via the same `aimPointOnArc()`-shaped helper in each class, so the marker position always matches where a shot fired at that instant would actually go.

Side guns fire from two side mounts instead of one front-center point (`GameScene.spawnPlayerProjectile`/`sideGunMountPosition()`, offset by the car's own half-width plus `SIDE_GUN_MOUNTS.extraOffsetPx` so the barrels visibly poke out past the body) — a single trigger pull spawns one projectile from each mount, aimed via `sideGunAngleDeg` (mirrored outward, see above), consuming the usual one ammo per pull (so the twin shot is "free" relative to the old single-shot ammo cost; `WEAPONS.sideguns.damage` was lowered from 18 to 10 per projectile to compensate for firing two at once).

### Shared weapon instability

Originally only the turret's spread reacted to handling (steering/off-road/drift). Per gameplay.md, all three weapons should get shakier together at speed and on rough terrain too, representing the car itself jolting the driver around rather than a per-weapon cause. `WeaponController.update()` (`weapons.ts`) computes one `sharedInstabilityDeg` value from `WeaponUpdateParams.speedFraction`/`onRoughTerrain` (both optional, defaulting to 0/`false` so existing callers/tests that don't pass them are unaffected) via `WEAPON_INSTABILITY` in `config.ts`, then folds it into all three weapons' existing mechanics rather than introducing a fourth parallel system:

- **Turret**: added directly into `targetSpread` alongside the existing steering/off-road additions, before the drift multiplier and smoothing — so it's just one more contributor to the spread the turret already had.
- **Side guns**: added as an unconditional jitter term on `sweepAngleDeg` (separate `deterministicJitter` call/seed offset from the existing steering-triggered jitter, so the two don't cancel out when both are active), scaled by `dtSeconds` like the existing jitter.
- **Rocket**: a wholly new `rocketSpreadDeg` state field (smoothed toward `sharedInstabilityDeg` the same way the turret smooths toward its own target, via `WEAPON_INSTABILITY.smoothing`), since the rocket previously had no instability mechanic of its own — `tryFire()` now rolls a `pseudoRandom`-based spread offset for it exactly like the turret already did, deflecting it off dead-ahead.

`PlayerCar.drive()` computes `speedFraction` (`|forwardSpeed| / PLAYER_HANDLING.maxForwardSpeed`, clamped to 1) and passes through `input.onRoughTerrain` into `weapons.update()` each frame, alongside the pre-existing steering/offRoad/drifting params.

## Health bars and high score

`HudSystem.drawHealthBars()` draws a small `Graphics` bar (`HEALTH_BAR` in `config.ts`) below the player (green) and above every active rival (red), proportional to `health / maxHealth` — a graphical supplement to the numeric health readout, following the same pattern as the weapon meter. Drawn fresh every frame (`g.clear()` then redraw) rather than persistent per-entity game objects, since rivals can be destroyed/despawned mid-race.

The player's bar specifically sits *below* the car rather than above: the area above (strictly, now, just ahead along heading) the player is used by `drawWeaponMeter()` for the equipped weapon's aim indicator (see Weapon visuals & in-world meter) — keeping the health bar on the opposite side avoids the two competing for the same space, the same reasoning that applied before this rework.

## Weapon sidebar

`WEAPON_SIDEBAR` (`config.ts`) positions a persistent sidebar pinned to the bottom-right corner of the screen (`x: 800, yStart: 400` on the 960×600 canvas, all sidebar objects `setScrollFactor(0)` so they stay fixed regardless of where the camera has scrolled to follow the player) — moved here from a mid-right position once the world became a scrolling 2D loop rather than a fixed vertical lane with a static dead-ground margin. `HudSystem`'s constructor builds one row per `WEAPON_IDS` entry: a static `Rectangle` swatch tinted with that weapon's `WEAPON_VISUALS` color (created once — the tint never changes, so no need to redraw it every frame), a number-key label, a name label (`WEAPON_LABELS`), and an ammo `Text` kept in `sidebarAmmoTexts` (keyed by `WeaponId`) so `updateWeaponSidebar()` can update just the ammo count each frame without touching the static parts.

`HudSystem.updateWeaponSidebar()` also draws a reload/readiness bar inline with each row's ammo count, on a dedicated `sidebarReloadBars` `Graphics` cleared and redrawn every frame: `pct = 1 - cooldownRemaining / WEAPONS[id].fireCooldown` (clamped to `[0, 1]`), filled in that weapon's `meterColor`, at `(WEAPON_SIDEBAR.x + 50, rowY + ...)` so it sits just to the right of the ammo text. This is the same value the rocket's old in-world meter showed, generalized to all three weapons in one place now that the rocket no longer has an in-world meter of its own (see Weapon visuals & in-world meter above) — side guns/turret already show their own aim state in-world (sweep arc / spread cone), so this bar is specifically about reload readiness, not aim.

A separate `Graphics` (`weaponSidebarHighlight`, depth `DEPTHS.hud - 1` so it sits behind the row's text/swatch) is cleared and redrawn each frame as a translucent rect (sized to cover through the reload bar) behind whichever row matches `player.weapons.current`, by index into `WEAPON_IDS` — the same array order the `InputHandler`'s `keydown-ONE/TWO/THREE` bindings use, so the sidebar's numbering always matches what the keys actually do.

This exists alongside, not instead of, the numeric `weaponText` HUD line in the corner — the sidebar's job is showing the *full roster* and how to reach each one at a glance, which a single "Weapon: X (ammo)" line can't do on its own.

Best distance is persisted across runs via `localStorage` (`HIGH_SCORE_STORAGE_KEY` in `GameScene.ts`) — read once in `create()` and passed into `HudSystem`'s constructor for the initial "Best: N m" line, compared/written in `endGame()`. The game-over overlay shows either the stored best or "New Best!" depending on whether the just-finished run beat it. This is the one piece of state that survives a `scene.restart()` (everything else resets in `create()`), and intentionally so — it's a record to chase, not meta-progression (see gameplay.md's Scoring & progression).

## Input

`src/systems/InputHandler.ts` owns keyboard key registration/polling and the weapon-select bindings; `src/systems/TouchControls.ts` owns the touch-only virtual joystick/fire button plus weapon-sidebar tap-to-switch and turret aim-pointer resolution (both run on mouse too — see Touch controls below). `GameScene.update()` OR-combines `InputHandler`'s booleans with `TouchControls.getMoveInput()`'s, the same way `InputHandler` already OR-combines WASD with the arrow keys — touch and keyboard are two alternate sources for the same `PlayerInput` booleans, not a separate input model.

- Movement: keyboard (WASD/arrows, W/Up accelerate, S/Down brake-then-reverse, A/Left and D/Right steer) polled via `Key.isDown`, or the virtual joystick on touch (digital thresholds against `MOBILE_CONTROLS.joystickDeadzoneX/Y`, not raw analog magnitude, so it drives the exact same booleans and `playerPhysics.ts` needs no changes) — plus Shift for drift, keyboard-only (no touch equivalent yet, see Touch controls below).
- Weapon select: keyboard one-shot `keydown-ONE/TWO/THREE` listeners (`InputHandler`), or tapping/clicking a weapon-sidebar row (`TouchControls`, both platforms) — both call `PlayerCar.selectWeapon()`.
- Fire: weapon-dependent, per gameplay.md — Space or the mobile fire button (hidden while turret is equipped) for rocket/side-guns; for the turret, whichever pointer `TouchControls.getTurretAimPointer()` resolves as the aim pointer this frame, held down and outside the UI.
- Restart: keyboard (Space) and pointer (tap/click) — dual-path as CLAUDE.md requires.

## Touch controls

`TouchControls` (constructed once in `GameScene.create()`, fed/read every frame from `update()`) is gated by `isMobileMode()` (`src/utils/device.ts` — touch-capability auto-detect via `navigator.maxTouchPoints`/`ontouchstart`, with a `?mobile=1`/`?mobile=0` URL override for testing without real touch hardware) for its *visual/joystick/fire-button* half only — the sidebar-tap and turret-pointer-resolution half run unconditionally on both platforms:

- **Virtual joystick** (mobile-only): a fixed base/thumb pair drawn via one screen-anchored (`setScrollFactor(0)`) `Graphics` object, base at `MOBILE_CONTROLS.joystickBaseX/Y` (bottom-left). On `pointerdown` within `joystickRadius` of the base, the touching pointer's `id` is recorded; subsequent `pointermove`s for that same `id` clamp the thumb offset to `joystickRadius` and store it as a normalized `{x, y}` vector (`Math.cos/sin(angle) * dist / radius`); matching `pointerup`/`pointerupoutside` clears it. `getMoveInput()` converts that vector to `accelerate`/`brake`/`left`/`right` via separate per-axis `joystickDeadzoneX`/`joystickDeadzoneY` thresholds (X deliberately higher than Y, so small sideways wobble while pushing straight up doesn't read as steering) rather than passing analog magnitude through — this is what lets `playerPhysics.computeDrive()` stay untouched by this whole feature.
- **Fire button** (mobile-only): a fixed circle at `MOBILE_CONTROLS.fireButtonX/Y` (bottom-right, positioned clear of the weapon sidebar's own footprint), tracked the same pointer-`id` way as the joystick. Not drawn, and excluded from `getTurretAimPointer()`'s UI check, while the turret is equipped, since the button does nothing for it — that screen area becomes tap-aimable like the rest of the field in that mode.
- **Weapon-sidebar tap-to-switch** (both platforms): `weaponSidebarRowRect(index, originX, originYStart)` (`config.ts`) is the one shared source of truth for each row's hit rect, used both by `HudSystem.updateWeaponSidebar()`'s selected-row highlight and by `TouchControls`' `pointerdown` hit-test, so a tap always lands exactly where the highlight is drawn. This also fixes a latent desktop quirk: clicking the sidebar used to do nothing *but* still register as "pointer is down" for the old unconditional turret-fire check below. The `originX`/`originYStart` params (resolved once via `sidebarOrigin()` — see Portrait-by-default on mobile below) are what let this same sidebar render correctly on both the desktop and mobile canvas sizes.
- **Turret aim-pointer resolution** (both platforms): `getTurretAimPointer()` scans `scene.input.manager.pointers` for the first one that `isDown` and isn't the joystick's/fire-button's tracked `id` or currently over the sidebar/joystick/fire-button zones. `GameScene.handleFiring()`'s turret branch and `HudSystem.drawWeaponMeter()`'s crosshair both resolve through this now instead of reading `scene.input.activePointer` directly — on desktop it's effectively "the mouse, unless it's over the sidebar"; on mobile it's "whichever non-joystick finger is down outside the UI," which is what lets a tap aim+fire the turret while the other thumb is still on the joystick. `main.ts`'s `input: { activePointers: 2 }` (see Runtime architecture above) is what makes that second simultaneous finger visible to Phaser at all.

## Portrait-by-default on mobile

Mobile uses a portrait (taller-than-wide) canvas by default; desktop stays the landscape 960×600 it always was. The tricky part wasn't picking a portrait size — it's that `isMobileMode()` (`utils/device.ts`) touches `window`/`navigator`, which makes it unsafe to call at `config.ts`'s own module-load time: Vitest's environment is plain Node (no `jsdom`/`happy-dom` configured — confirmed via `vite.config.ts`), and `config.ts` is transitively imported by nearly every unit test, so any top-level `isMobileMode()` call there would crash the whole suite on import.

The resolution keeps `config.ts` itself Node-safe by never calling `isMobileMode()` there at all:

- `CANVAS_WIDTH`/`CANVAS_HEIGHT` (960/600) stay exactly what they were — a plain static desktop default. A second pair, `MOBILE_CANVAS_WIDTH`/`MOBILE_CANVAS_HEIGHT` (600×960, a clean inverse ratio — not an attempt to match any specific phone's exact aspect, `Scale.FIT` letterboxes any mismatch anyway), sits alongside it, also just plain numbers.
- **`main.ts`** — never imported by Vitest, since it's the Vite app entry point, not part of any test's import graph — is the *only* place that calls `isMobileMode()`, to pick which pair to actually pass into the `Phaser.Game` config's `width`/`height`.
- Everywhere else that used to import the static `CANVAS_WIDTH`/`CANVAS_HEIGHT` purely for *layout* (not for the Game config itself) reads the scene's own `this.scale.width`/`this.scale.height` instead — Phaser's `ScaleManager` always reflects whichever size `main.ts` actually chose, with no risk of a stale baked-in value. This is `GameScene`'s `uiCamera` creation and game-over/finish overlay, and `IntroScene`'s background/text positions.
- `WEAPON_SIDEBAR` (`config.ts`) renders on *both* canvas sizes, so it can't bake in either one: it stores `rightMargin`/`bottomMargin` instead of absolute `x`/`yStart`, and a new pure helper, `sidebarOrigin(canvasWidth, canvasHeight)`, resolves those margins against whatever size is actually live. `HudSystem` and `TouchControls` each call it once in their constructor (`scene.scale.width/height`) and cache the result (`sidebarX`/`sidebarYStart` instance fields) — the canvas size never changes mid-session, so there's no need to re-resolve every frame.
- `MOBILE_CONTROLS` (joystick/fire-button positions), by contrast, stayed plain absolute numbers — everything in it only ever renders when `isMobileMode()` is true, at which point the canvas is *always* exactly `MOBILE_CANVAS_WIDTH`×`MOBILE_CANVAS_HEIGHT`, so there's nothing to resolve dynamically. They were re-tuned for that 600×960 canvas rather than the old 960×600 assumption — the fire button in particular moved from a now-irrelevant `700` to a bottom-*center*-right position clear of both the joystick and the now-narrower sidebar's footprint, rather than competing with the sidebar for the literal bottom-right corner.

`playerPhysics.ts`/`weapons.ts`/`enemyBehaviors.ts`/`track.ts` (the pure, unit-tested modules) never referenced canvas dimensions before this and still don't — track/road generation is entirely world-space, independent of whatever the camera happens to be looking through.

## Camera look-ahead

`GameScene.update()`'s `updateCameraLookAhead()` biases `cameras.main`'s follow target toward wherever the player is currently heading, scaled by speed fraction (`CAMERA.lookAheadMaxPx` at full speed, 0 near-stationary) and smoothed toward that target each frame (`CAMERA.lookAheadLerp`, converted to a delta-normalized ease factor so the smoothing rate doesn't depend on frame rate, consistent with everything else here being driven off `delta`) rather than snapping. Applied via `cameras.main.setFollowOffset(x, y)`, which Phaser resolves by centering the view on `(target.x - x, target.y - y)` — so the offset passed in is the *negative* of the desired look-ahead vector; shifting the camera's center ahead of the player puts more of the upcoming road in view, with the player riding nearer the trailing edge of the screen instead of dead-center. Zoom (`cameras.main.setZoom()`, set once in `create()`) is `CAMERA.zoom` normally, `CAMERA.mobileZoom` (zoomed out further) when `isMobileMode()` — a smaller/touch-occluded screen needs more margin to read an upcoming bend in time. Both apply on desktop and mobile alike; only the zoom *amount* differs by platform.

## UI camera (screen-anchored HUD/touch controls vs. camera zoom)

Adding zoom to `cameras.main` above broke every screen-anchored HUD element and the new virtual joystick/fire button: `setScrollFactor(0)` only cancels camera *scroll*, not *zoom* — a scrollFactor(0) object's render position still gets pulled toward/away from the viewport center by the zoom factor, while pointer hit-testing (`pointer.x/y`) stays true 1:1 screen space regardless of zoom. The practical symptom was exactly "the joystick renders in one place, but tapping there does nothing" (see `docs/troubleshooting.md` for the full investigation, including why a naive synthetic-touch test didn't catch it).

The fix is a second camera, `GameScene.uiCamera` (`this.cameras.add(0, 0, this.scale.width, this.scale.height)`, created first thing in `create()`, left at its default zoom 1/scroll (0,0)) dedicated to anything that must render at a true screen position regardless of `cameras.main`'s zoom/scroll. Getting one camera to show only "world" content and the other to show only "UI" content needs an exclusion relationship on *both* sides, handled two different ways depending on which side is more enumerable:

- **Every world entity excludes itself from `uiCamera`** at construction time, via `ignoreInUiCamera(scene, this)` (`src/utils/cameraLayers.ts`, duck-typed off `scene.uiCamera` to avoid entities importing `GameScene`'s class) — called in `PlayerCar`/`EnemyCar`/`Projectile`/`Hazard`/`Pickup`'s constructors (right after `scene.add.existing(this)`), plus inline in `GameScene.drawTrack()`'s track `Graphics` and `spawnExplosion()`'s fx image. This has to happen *per-object at construction*, not as a one-off `camera.ignore(group)` call in `create()` — the projectile/hazard/pickup pools grow lazily over the course of a race (`Group.get()` only constructs a new instance the first time a pool slot is actually needed), so a one-off call would miss anything built later. `HudSystem.drawWeaponMeter()`'s `weaponMeter`/`healthBars` graphics are the one exception *within* `HudSystem` — they float in world space near cars (no scrollFactor override), so they call `ignoreInUiCamera` too despite living in the "UI" class.
- **`cameras.main` excludes the UI objects** instead, in one call each right after they're constructed: `cameras.main.ignore(hud.getUiLayer())` and `cameras.main.ignore(touchControls.getDisplayObjects())` in `GameScene.create()`, plus `cameras.main.ignore(this.overlay)` once the game-over/finish overlay is built. `HudSystem` collects every screen-anchored object it creates (text readouts, weapon sidebar, sidebar highlight/reload-bar graphics) into one `scene.add.layer()` (`getUiLayer()`) specifically so this is a single call instead of enumerating ~20 individual objects — a `Layer`'s own ignore-state gates whether the renderer recurses into its children at all, the same way a `Container`'s does, so children added to it later (there are none here — HudSystem builds everything once in its constructor) would still need the layer itself ignored, not each child individually.

## Testing

Vitest is wired up (`npm test`), across five files, all pure-logic:

- `playerPhysics.test.ts` — handling math: accel/brake/reverse, heading-based speed-scaled steering (including the reverse-flips-turn-direction behavior), drift (faster turning + velocity-direction slip lagging behind heading, vs. fast/near-instant catch-up while gripping), the temporary max-speed override used by the speed-boost pickup, the two terrain effects (extra drag on rough terrain; reduced turn authority + a velocity-heading bias while oil-slicked, including that a zero/flipped bias contributes nothing/the opposite push), and canyon-wall contact (extra drag + the `atWall` flag, derived from `lateralOffset` vs. `leftWallDist`/`rightWallDist` rather than world x bounds).
- `weapons.test.ts` — `WeaponController`: ammo consumption/cooldown gating, ammo refill clamped to max, side-gun sweep bounds, turret instability rising with steering/off-road/drift, the shared speed/rough-terrain instability reaching the turret and side-gun sweep, and the rocket firing dead-ahead (`angleDeg === 0`) regardless of speed/terrain — it's the one weapon explicitly exempt from the shared instability. Unaffected by the heading-based steering rewrite — this module only ever dealt in relative angles, never positions/heading.
- `enemyBehaviors.test.ts` — `computeTargetHeading`: follows the track lookahead when the player is far away, blends strongly toward the player when close (chasesPlayer archetypes), Shooter steering away when too close vs. just following the track otherwise, Heavy ignoring the player entirely, avoidance ignored beyond the avoidance radius, a high-avoidanceWeight archetype steering hard away from a close rival vs. Heavy barely reacting, and an aggressive archetype only weakly avoiding (pulled down but still net-toward-the-player); `turnTowardHeading`: rate-limited toward a target (capped at maxDeltaDeg, snaps if already within range, turns via the shortest rotational direction); `curvatureSpeedFactor`: full speed straight ahead, dropping toward (never below) `minSpeedFactor` as the upcoming-bend angle widens.
- `track.test.ts` — `generateTrack`/`pointAt`/`nearestPoint`/`wallDistancesAt`: a generated track forms a non-trivial closed loop (samples wrap back to the start via `pointAt`, all within the computed bounds), `nearestPoint` reports ~zero offset/distance exactly on the centerline and the correct sign to either side of the direction of travel, `wallDistancesAt` keeps each side's distance within `[pavedHalfWidth + minOffset, pavedHalfWidth + maxOffset]` and is periodic without left/right moving in lockstep, and `nearestPoint`'s `hintS` continuity behavior specifically (see Track & rivals above): without a hint the seam can resolve to either `s=0` or `s=totalLength`, with a hint it resolves to whichever is within half a lap of it, and tracking forward progress across the seam via successive hinted calls shows a small continuous increase rather than a jump.
- `config.test.ts` — the pure helper functions that live in `config.ts` alongside the tunables they consume rather than in a `*Car`/system module: `wallImpactDamage` (no damage at/below `minImpactSpeed`, linear scaling, capped at `maxDamage` past max speed) and `nextCrateIntervalMs` (stays within `intervalMs ± intervalJitterMs` across the rng's full `[0, 1]` range — see Pickups above).
- `EnemyCar`'s own `drive()` (turn-rate application, speed formula, ram recoil) isn't unit-tested directly — it's a thin Phaser wrapper around the tested `enemyBehaviors` functions, in the same spirit as `PlayerCar` around `playerPhysics`, and covered by the e2e suite/manual playtesting instead (see below). The `systems/` classes are similarly untested directly where they're thin Phaser wrappers around already-tested logic or scene-level glue (`HudSystem`, `InputHandler`); `CollisionHandler` and `RaceTracker`'s actual rules are exercised indirectly through the e2e suite rather than unit-tested in isolation, since both depend on `EnemyCar`/`PlayerCar`/`Track` instances rather than being Phaser-free the way `playerPhysics`/`enemyBehaviors`/`track` are.
- `playerPhysics.test.ts` predates the others; new pure modules followed the same pattern immediately rather than being left untested.

Manual `npm run dev` playtesting (or the headless-Chromium `playtest` skill, for one-off scripted/visual checks during a single work session) remains how purely visual things get verified — pixel alignment, color/contrast, "does this read as intended" — `window.__GAME_STATE__` numbers wouldn't have caught the sidegun-meter alignment bug below, since the bug was specifically that the *drawn* position didn't match the *real* mount position; both computed "correctly" by their own (different) math. Anything that reduces to checkable game state, though, now has a real regression suite instead of being re-verified by hand or by a throwaway script every time:

### E2E (Playwright)

`e2e/` is a separate Playwright Test project from Vitest's unit tests — `npm run test:e2e` (`playwright test --config e2e/playwright.config.ts`), against the real `npm run dev` server (`baseURL` in `e2e/playwright.config.ts`, defaults to `localhost:5173`). It needs its own config pointed at explicitly: running plain `playwright test` from the repo root with no `--config` flag picks up Vitest's root-level `*.test.ts` files too (Playwright Test's default `testMatch` matches `.test.ts` as well as `.spec.ts`) and fails immediately trying to run `describe`/`it` through Playwright's test runner; the inverse problem — Vitest picking up `e2e/tests/*.spec.ts` and failing on Playwright's `test()` API — is why `vite.config.ts`'s `test.exclude` explicitly excludes `e2e/**` (Vitest's own default test-file matcher would otherwise find it too). Each suite only runs its own files.

**The `window.__GAME_STATE__` hook** (`GameScene.updateE2EGameState()`) is how tests assert on live game state without scraping canvas pixels or duplicating game logic in the test file. It's gated behind `window.__E2E_TEST__` (set when `?e2e=1` is in the URL, or pre-set via `page.addInitScript` before navigation) so it never runs — or costs anything — during ordinary play; `create()` calls it once synchronously (so state exists immediately, before the first `update()` tick) and `update()` refreshes it every frame after that. The snapshot is deliberately minimal: player position/health/laps/speed/weapon/ammo/live race position, active rivals' position/health/laps, hazard counts by type, `gameOver`/`won`, and the same `raceDebugText` string shown on screen — enough to assert real outcomes without coupling tests to internals (e.g. tests check `player.laps`, not `this.playerStartS`).

**`?seed=` makes a race reproducible.** `GameScene.getSeedFromUrl()` reads it and, if present and numeric, wraps it in `track.ts`'s `createSeededRng()` (a small LCG, also used directly by `track.test.ts` for the same reason) to build `this.rng`. Every `Math.random()` call that used to be scattered through `GameScene` (`generateTrack`, archetype picks, hazard shuffle/placement jitter, rubber-band activation rolls, oil-slick bias rolls) goes through `this.rng` instead, so a seeded race is reproducible end to end — not just the track shape. Position-based assertions still need a tolerance, not exact equality (`e2e/tests/core.spec.ts`'s same-seed test): two separate page loads reaching their "ready" signal don't tick the same number of real-time-driven physics frames, and rivals are actively AI-driving at ~500-600px/s the whole time (unlike the near-idle player), so a frame or two of difference compounds into tens of px for a rival, while hazard placement (no per-frame movement) matches exactly.

`e2e/tests/core.spec.ts` covers: the player accelerating, the lap/position HUD fields existing, **a regression test for the start/finish-seam phantom-lap bug** (see Track & rivals above — drive nothing, assert `laps === 0` at race start), exactly 5 rivals spawning with health/laps, hazards of all three types existing, live race position staying in range, weapon switching, ammo consuming on fire, and the seed reproducing (same seed) vs. not reproducing (different seed) the track/rival layout.

Phaser scene wiring, rendering, and input plumbing that *doesn't* reduce to checkable state — visual alignment, color, layout — stays manual (`npm run dev` or the `playtest` skill) per CLAUDE.md; not worth fighting Phaser's runtime for pixel-level assertions.
