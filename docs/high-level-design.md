# Hit the Road — High-Level Design

Architecture/tech reference for the project. For intended gameplay behavior see [gameplay.md](gameplay.md); for contribution conventions see [CLAUDE.md](../CLAUDE.md) (not duplicated here).

## Tech stack

- **Language**: TypeScript 5.5 (`strict`, `noUnusedLocals`/`noUnusedParameters`, ES2020 target, `bundler` module resolution, no emit — Vite handles bundling)
- **Build/dev tool**: Vite 5.4 (`npm run dev` / `build` / `preview`)
- **Game framework**: Phaser 3.80 — rendering (Canvas/WebGL via `Phaser.AUTO`) and Arcade Physics for movement/collision
- No other runtime dependencies.

## Project structure

```text
src/
  main.ts                    — Phaser.Game config + bootstrap
  config.ts                   — tunable gameplay numbers (handling, road geometry, archetypes, weapons, pickups, hazards)
  entities/
    PlayerCar.ts               — Phaser wrapper: health, drift, weapon selection/firing, applies handling to the sprite
    playerPhysics.ts            — pure handling math (accel/brake/reverse/steer/off-road drag/drift), Phaser-free
    playerPhysics.test.ts
    weapons.ts                  — WeaponController: pure ammo/cooldown/sweep-meter/turret-spread logic, Phaser-free
    weapons.test.ts
    EnemyCar.ts                 — Phaser wrapper: archetype, health, AI-driven velocity (with steering inertia + traffic avoidance), fire timer, ram cooldown
    enemyBehaviors.ts            — pure per-archetype lateral-AI function (target velocity + avoidance blend) and a steering-smoothing helper, Phaser-free
    enemyBehaviors.test.ts
    Projectile.ts                — pooled player/enemy bullet & rocket sprite
    Pickup.ts                    — pooled health/ammo/boost drop
    Hazard.ts                    — pooled rough-terrain/oil-slick road patch (continuous effect via an explicit per-frame overlap check, not a one-shot obstacle)
  scenes/
    BootScene.ts                — loads art/audio assets, procedurally draws road/divider, hands off to IntroScene
    IntroScene.ts                — one-time controls/orientation screen, hands off to GameScene
    GameScene.ts                 — orchestrates spawning, combat, scoring, input wiring, HUD, game over
public/assets/
  cars/, fx/, hazards/, audio/  — loaded CC0 assets (see CREDITS.md), served as static files by Vite
scripts/
  process-car-sprites.mjs        — one-off-but-repeatable preprocessing for the source car pack (rotate/resize)
  setup-headless-chromium.sh     — headless-browser environment setup (see .claude/skills/playtest/)
docs/
  high-level-design.md — this file
  gameplay.md           — gameplay/feature specification
  troubleshooting.md    — environment gotchas and their fixes, recorded as hit
```

`src/entities/` and `src/config.ts` now hold the full gameplay layer: movement, weapons, enemy AI, pickups, and hazards. The Phaser-wrapper/pure-logic split established for `PlayerCar`/`playerPhysics` was carried through to `weapons.ts` (`WeaponController`) and `enemyBehaviors.ts` (`computeLateralVelocity`) — anything that's actually a rule (math, state transitions) lives in a plain, Phaser-free module with its own unit tests; the `*Car`/`Projectile`/`Pickup`/`Hazard` classes are thin wrappers that own a sprite/physics body and call into that logic. `Projectile`, `Pickup`, and `Hazard` don't have an equivalent pure-logic split since they don't have any non-trivial rules of their own (they're driven entirely by config + the systems above).

## Runtime architecture

- One `Phaser.Game` instance (`main.ts`), fixed 480×720 canvas (`CANVAS_WIDTH`/`CANVAS_HEIGHT` in `config.ts`), Arcade Physics enabled, scene list `[BootScene, IntroScene, GameScene]`.
- **BootScene**: `preload()` loads cars/fx/hazards/audio from `public/assets/` and procedurally draws only the road/divider textures (see Rendering & assets). Since `this.load.*` calls queue into Phaser's loader, `create()` no longer needs the manual async handling the earlier audio-synthesis version required — `scene.start("intro")` runs once the load queue completes, same as a synchronous `preload()` would.
- **IntroScene**: a static controls/orientation screen (key/action pairs as plain text, no game state) that starts `"game"` on `keydown-SPACE` or `pointerdown`, the same dual-path restart pattern `GameScene`'s game-over overlay uses. Only reached once per page load — `GameScene.endGame()`'s restart calls `this.scene.restart()` directly on `GameScene`, bypassing `IntroScene` entirely, since a player who just died already knows the controls.
- **GameScene**: orchestrates —
  - a `PlayerCar` driven by continuous WASD/arrow input (accelerate/brake-then-reverse/steer), Shift to drift, 1/2/3 to switch weapons, Space/click to fire,
  - scrolling road divider via `TileSprite` offset, tied to the player's own forward speed,
  - timed `EnemyCar` spawning, archetype chosen by a score-gated weighted pick (`ENEMY_UNLOCKS`) which also determines where it appears — "behind" archetypes spawn at a random x off the bottom of the screen, "side" archetypes spawn at the road's left/right edge within a band just ahead of the player's current y — with spawn interval and enemy approach speed scaling with distance, gated by a hard `ENEMY_SPAWN.maxConcurrent` cap checked before every spawn attempt so the road stays a handful of readable opponents rather than a swarm,
  - timed `Hazard` spawning (rough-terrain/oil-slick road patches) on the same distance-scaling pattern as enemies, plus a separate timed standalone health/ammo crate spawn (`CRATE_SPAWN`) independent of kills,
  - three `physics.add.overlap` pairs (player↔enemy-projectiles, player-projectiles↔enemies, player↔pickups) plus two `physics.add.collider` pairs (player↔enemies and enemies↔enemies, both for physical contact — see Entity architecture and Collision shunt) — hazard contact is deliberately *not* a registered pair at all; see Terrain hazards for why it's checked explicitly instead,
  - a text HUD (health, distance, best distance, speed, weapon+ammo, and a per-weapon aim/cooldown line) plus graphical overlays (per-weapon aim meter above the player, health bars below the player and above every active enemy) updated every frame,
  - game over → overlay → restart (space/tap or scene restart) once player health reaches zero, unchanged in shape from the original instant-crash version, just gated on health now instead of any contact; the overlay also reads/writes a `localStorage`-persisted best distance (see Health bars and high score).

## Entity architecture: composition over deep inheritance

Built out once a second archetype actually needed different behavior, as planned:

- `EnemyArchetypeConfig` (in `config.ts`) holds each archetype's stats (health, speed multiplier, lateral speed, fire stats, score value) plus behavior flags (`chasesPlayer`, `keepsDistance`) and where it spawns (`approachFrom: "behind" | "side"`) — adding a 5th archetype is a new config entry, not a new class. Contact sound/fx is no longer archetype-specific (the old `explodesOnContact` flag was removed) — it's now derived purely from whether the contact actually killed the enemy (explosion) or not (plain collision), the same rule `handleEnemyHitByProjectile` already used for weapon kills.
- `enemyBehaviors.computeLateralVelocity()` is the swappable movement strategy: a pure function reading those flags to decide an enemy's *target* lateral velocity relative to the player, then blending in an avoidance term (steer away from whichever other enemy is currently closest, within `ENEMY_AI.avoidanceRadius`) weighted by how aggressive the archetype is — `chasesPlayer` archetypes barely avoid (`aggressiveAvoidanceWeight`), others avoid much harder (`passiveAvoidanceWeight`). `enemyBehaviors.smoothLateralVelocity()` then eases the car's actual velocity toward that target frame over frame (inertia) rather than snapping to it. `EnemyCar.drive()` calls both for the x-axis and, for the y-axis, applies a single shared formula — own distance-scaled speed vs. the player's current forward speed — regardless of `approachFrom`; only the spawn position (`GameScene.spawnEnemy()`) differs between "behind" and "side". Neither AI rule has a Phaser dependency; both are unit-tested directly. `GameScene.updateEnemies()` computes each enemy's nearest rival (by Euclidean distance) once per frame and passes it into `drive()`.
- Enemies also physically collide with each other via a `physics.add.collider(enemies, enemies)` pair (`GameScene.create()`) — the avoidance steering above reduces how often this triggers but doesn't prevent it, especially for aggressive archetypes that barely try to avoid in the first place. The player↔enemy pairing is a `physics.add.collider` too, for the same reason — see Collision shunt below for why that one specifically used to be an `overlap` and had to change.
- `weapons.WeaponController` plays the equivalent role for the player's weapons: ammo, cooldown, the side-gun sweep meter, and turret instability are all pure state transitions in one class, parameterized by `WeaponConfig`/`SIDE_GUN_SWEEP`/`TURRET_STABILITY` in `config.ts`. `PlayerCar` holds one instance and forwards handling state (steering/off-road/drifting) into it each frame; it doesn't know how any individual weapon works.
- This avoided a combinatorial subclass tree for either enemies or weapons — four enemy archetypes and three weapons exist as data + two small behavior modules, not seven classes.

## Pooling

All spawned-and-despawned object types are pooled Arcade Physics groups with a `maxSize` (enemies, player projectiles, enemy projectiles, pickups, hazards) — `group.get()` reuses an inactive instance (or creates one up to `maxSize`), and each entity has a `spawn()`/`despawn()` pair that resets its state instead of `destroy()`/recreate. This applies from the start for all of these types, per the "projectiles should be pooled from the start" and "pool once a second archetype with non-trivial reset state ships" guidance recorded here previously — both conditions are now true at once (weapons + three more archetypes landed together), so there was no reason to stage it.

One-off visual effects (the explosion sprite on a kill) are *not* pooled — they're rare relative to projectiles/enemies, created with a tween that self-destroys, and not worth the bookkeeping.

## Collision shunt and ram damage (ramming no longer costs the player health, but now costs the enemy)

`GameScene.handlePlayerEnemyCollision()` used to look up a flat `ramDamage`/`explosionDamage` number on the archetype and subtract it from player health, then destroy the enemy unconditionally on any contact — that unconditional destroy was the bug behind enemies "exploding on a single touch" regardless of their health pool. Ramming now only knocks the player's speed around (no player health cost) — `COLLISION_SHUNT` (`config.ts`) converts the *actual relative closing speed* of the impact (`Math.abs(enemy.body.velocity.y)`, the same quantity `EnemyCar.drive()` already computes as `forwardSpeed - ownSpeed`) into a magnitude, floored at `COLLISION_SHUNT.minShunt` so even a near-zero-relative-speed bump still registers as something.

That magnitude is applied via `PlayerCar.applyImpactShunt(signedAmount)`, which adds a signed delta directly to `forwardSpeed` — it would do nothing useful to dampen-toward-zero logic for a player sitting still (a stationary player rammed by a fast chaser should still visibly lurch). Direction depends on `approachFrom`: "behind" archetypes ram from the rear and jolt the player forward (positive), "side" archetypes clip them from alongside and jolt them back (negative). The next frame's normal drive-speed clamp folds the result back into the valid range, so no separate clamping is needed here.

The *same* magnitude, scaled by `COLLISION_SHUNT.ramDamageFactor`, is now also applied to the enemy via its normal `takeDamage()` — ramming is just another damage source feeding the same health pool as weapon fire, not a separate instant-kill rule. The enemy is only destroyed (score, pickup roll, explosion fx/sfx, despawn — same `destroyEnemy()` path as a weapon kill) if that brings its health to zero; otherwise it just plays the plain collision sound, calls `enemy.applyDamageSlow()` (see Damage-slow below), and keeps driving, now hurt and temporarily slower. `EnemyCar.ramCooldown` (set to `COLLISION_SHUNT.ramCooldownMs` on every ram, decremented in `drive()`) guards `handlePlayerEnemyCollision()` against re-applying ram damage on the next frame or two of continued overlap. Hazards no longer cost health at all (see Road & hazards), and off-road driving and enemy gunfire are unaffected and still cost health normally.

**Player↔enemy contact had to become a `physics.add.collider`, not an `overlap`.** The original pairing was an `overlap` because the old code destroyed the enemy unconditionally on any contact, so there was never a second frame of contact to worry about. Once ramming started doing scaled damage instead of an instant kill, a surviving enemy would just keep moving at its own AI-driven velocity — unaffected by the "collision" — and glide straight through the player's fixed screen position with nothing to stop it, since an `overlap` only reports contact, it never separates the bodies. Switching to a `collider` makes Arcade Physics physically push the two bodies apart every step they touch, on top of still invoking the same ram-damage callback. One side effect: that separation can nudge `player.y` away from its intended fixed value (the player's sprite is meant to stay parked at `PLAYER_Y` while the world scrolls past it — see `GameScene.update()`'s `this.player.setY(PLAYER_Y)` at the top of every frame, the standard Phaser idiom for re-pinning an axis a collider might otherwise drift).

**The collider alone wasn't enough to make the bounce-apart feel reliable, though.** Pinning `player.y` every frame means the collider's *positional* separation along that axis gets undone the instant it happens for "behind" archetypes (the common case) — the only way the gap can actually reopen is for the *enemy* to move away under its own steam. But a chasing archetype's AI doesn't know it just rammed; left alone, it would simply recompute the same closing velocity next frame and get driven straight back into contact, fighting the collider every step — sometimes the collider "won" a frame (visible bounce) and sometimes the AI "won" (lingering overlap), which is exactly the inconsistency reported in playtesting. The fix is in `EnemyCar.drive()`: while `ramCooldown > 0`, it skips its normal AI entirely and instead moves directly away from the player at a fixed `COLLISION_SHUNT.recoilSpeed` ("behind" archetypes fall back in y; "side" archetypes peel off in x, away from the player), for the same `ramCooldownMs` window the damage-reapplication guard already uses. That guarantees the gap actually opens every time, rather than depending on how the collider and the AI happened to interact that particular frame.

## Speed parity and damage-slow

Enemy approach speed went through two rebalances in opposite directions before landing here. First pass: enemies could be categorically *faster* than the player at high distance (`ENEMY_SPAWN.maxApproachSpeed` × an aggressive archetype's `speedMultiplier` exceeded `PLAYER_HANDLING.maxForwardSpeed` by a wide margin), making outrunning anything impossible by definition once distance climbed far enough — fixed by capping `maxApproachSpeed` well under the player's max. That overcorrected: with both `baseApproachSpeed` and `maxApproachSpeed` lowered, enemies felt categorically *slower* throughout the whole run, not just at the previously-broken high end. The current values (`baseApproachSpeed: 260`, `maxApproachSpeed: 580`, chaser `1.05`/bomber `1.1`/shooter `0.85`/heavy `0.6`) land enemy top speed at roughly the player's own `maxForwardSpeed` (620) — bomber just over it, chaser just under — so neither side has a structural speed advantage; heavy and shooter stay deliberately slower as part of their archetype identity (see Enemies in gameplay.md), not as a general rule.

With speed roughly symmetric, the thing that actually decides a chase needed to be combat, not the spawn config — that's `DAMAGE_SLOW` (`config.ts`). Any discrete hit that *doesn't* kill its target — a weapon hit or ram-damage hit landing on an enemy that survives, or an enemy projectile hitting the player — temporarily caps the victim's top speed (`maxSpeedFactor`) for `durationMs`, on top of an immediate one-time knock to the player's *current* speed specifically (`speedMultiplier`, applied once in `PlayerCar.applyDamageSlow()` since the player has a persistent `forwardSpeed` value to multiply; enemies don't — their "speed" is recomputed fresh from config every frame in `EnemyCar.drive()`, so the timer-driven cap alone is the whole effect there). `GameScene` calls `applyDamageSlow()` from exactly three places: `damagePlayer()` (enemy gunfire hitting the player — *not* the player's own off-road drain, which calls `player.takeDamage()` directly from `PlayerCar.drive()` and bypasses `damagePlayer()` entirely, since a one-time speed multiply applied every single frame of continuous off-road drain would compound toward zero almost instantly), the non-lethal branch of `handleEnemyHitByProjectile()`, and the non-lethal branch of `handlePlayerEnemyCollision()`'s ram-damage path.

`PlayerCar`'s `damageSlowTimer` and `speedBoostTimer` (from the speed-boost pickup) both resolve to a `maxForwardSpeedOverride` passed into `computeDrive()` — if both are active at once, the damage slow wins, since getting hit should always feel impactful regardless of buffs. `EnemyCar.drive()` decrements `damageSlowTimer` unconditionally at the top, before the ram-cooldown recoil branch's early return, so it keeps counting down even during a recoil window rather than effectively pausing (which would make the visible debuff outlast `durationMs` by however long the recoil happened to run).

## Terrain hazards (rough road, oil slicks)

`Hazard` (one pooled group, `HazardType: "rough" | "oil"`) no longer despawns on contact the way the old debris/barrier obstacles did — it's a patch the player (and enemies) drive over, and it keeps affecting whoever's on it every frame, then scrolls off-screen naturally like any other world prop (`GameScene.updateWorldProps`). Textures are irregular polygon blobs (`BootScene.drawIrregularBlob()` — N points around a center at a randomized fraction of a base radius, filled as one shape) rather than a plain rectangle/ellipse, so they read as organic damage/spill rather than a geometric primitive.

Hazard contact is checked via an *immediate* `this.physics.overlap(...)` call inside `update()`/`updateEnemies()` rather than a `physics.add.collider`/`overlap` pair registered once in `create()`. This matters specifically here (and nowhere else in the scene) because the result feeds back into the *same frame's* `PlayerInput` before `player.drive()` runs — a registered pair's callback timing relative to our own `update()` isn't something we want this effect's correctness to depend on, since calling `physics.overlap()` ourselves at a known point removes that dependency entirely instead of reasoning about it.

- **Player**: `this.onRoughTerrain` is reset to `false`, then the explicit overlap check sets it back to `true` if still on a rough patch, all before building that frame's `PlayerInput` — so `computeDrive()` (`playerPhysics.ts`) sees a value from *this* frame, not a frame-old one. It applies extra drag (`ROUGH_TERRAIN.dragPerSecond`) whenever true, the same `decayTowardZero` mechanism off-road drag already uses. `this.oilSlickTimer` is refreshed to `OIL_SLICK.effectDurationMs` by the same check every frame still overlapping an oil patch, then counted down once the player leaves it, so the effect lingers briefly after exit. `computeDrive()` reduces turn authority (`OIL_SLICK.controlMultiplier`) and adds `input.oilDriftBias * OIL_SLICK.driftStrength` to lateral velocity whenever `oilSlicked` is true — turns less responsively *and* gets pushed sideways, "harder to control," not just "turns slower." `oilDriftBias` (a value in `[-1, 1]`) is rolled once by `GameScene` when `oilSlickTimer` transitions from inactive to active (`this.oilDriftBias = Math.random() * 2 - 1`), not every frame — an earlier version re-rolled it every frame via a fast-oscillating deterministic function of elapsed time, which playtesting found averaged out to a vibration-in-place rather than reading as a slide. Holding one rolled value for the whole activation makes the car visibly drift/slide in one direction for the duration instead, the way losing traction actually feels. `computeDrive` stays a pure, testable function since the bias is just a plain number passed in via `PlayerDriveInput`, not derived from any internal clock.
- **Enemies**: `GameScene.updateEnemies()` resets every active enemy's `onRoughTerrainThisFrame` to `false`, then runs one `physics.overlap(enemies, hazards, ...)` call per frame covering all enemies at once, setting `onRoughTerrainThisFrame`/`oilSlickTimer` on whichever ones are actually overlapping (rolling a fresh `EnemyCar.oilDriftBias` only on the same inactive→active transition the player's path uses). `EnemyCar.drive()` applies the matching effects directly (no pure-module involvement, unlike the player's path) — rough terrain scales down that frame's own approach speed by `ROUGH_TERRAIN.enemySpeedMultiplier`; an active oil-slick timer adds `oilDriftBias * OIL_SLICK.enemyDriftStrength` on top of (not folded into) the smoothed steering velocity, so it reads as a sudden loss of control rather than something the steering inertia would otherwise smooth away.

Neither hazard type costs health for player or enemies — that's the off-road/enemy-fire/hazard split called out in Collision shunt above.

## Rendering & assets

Mixed, deliberately: loaded CC0 assets for anything where sourced art is a clear win, procedural generation kept for the road/ground specifically to control palette consistency (see CLAUDE.md's Assets section for the policy this implements).

Render depth is centralized in `DEPTHS` (`config.ts`) and applied once per type in each entity's constructor (`EnemyCar`, `Hazard`, `Pickup`, `Projectile`, `PlayerCar`), rather than left at Phaser's default of 0 for everything. Without explicit values, same-depth objects tie-break by display-list insertion order, which depends on pool creation order — that's what let hazard patches render in front of cars depending on spawn timing. Ground decals (`hazard`, `pickup`) sit below `enemy`/`player`; `projectile` sits above both so a shot flying over a car right before impact stays visible.

- **Cars** (`public/assets/cars/`): FieraRyan's CC0 "Futuristic Wasteland Top-down Cars" — 5 of the pack's 9 designs, picked for visual variety, preprocessed once via `scripts/process-car-sprites.mjs` (rotated from the source's landscape orientation to portrait, flipped for archetypes that need to visually face the player, downscaled from ~1170px to 90px tall — shrunk from an initial 128px pass once cars on-screen read as too large/cluttered). `EnemyCar.spawn()` applies a per-archetype `tint` (in `ENEMY_ARCHETYPES`) since the source pack has no color variants of its own — the player car is left untinted. Both `PlayerCar` and `EnemyCar` additionally apply `CAR_SCALE` (0.6) in their constructors, on top of that preprocessing, to leave more visible road on the fixed-size canvas.
- **Fx** (`public/assets/fx/`): Kenney's Particle Pack (explosion/projectile/pickup glow shapes — these are grayscale by design, meant to be recolored) plus an ammo-pickup icon from Kenney's Racing Pack. `GameScene` applies `VISUAL_TINTS` (in `config.ts`) plus `Phaser.BlendModes.ADD` for the glow-style fx at each spawn site, both to recolor Kenney's brighter house palette toward this game's grimmer one and, for the particle shapes, to get the intended glow look. Player projectiles specifically use `WEAPON_VISUALS` (tint + scale per weapon) instead of the shared fx tints, so the three weapons stay visually distinct from each other (see Weapons below).
- **Road/ground/terrain hazards**: still `Graphics.generateTexture()` in `BootScene`, not sourced — Kenney's matching road tiles read as bright kart-racer colors (light-blue track, candy-orange stripes) that clash badly with the gritty cars, and a flat recolor of our own simple rectangles was both consistent and cheap. `ROAD_COLORS` in `config.ts` holds the road/ground palette (dark cracked-asphalt road, dead-ground margins, dirty-yellow divider); `TERRAIN_COLORS` holds the rough-road/oil-slick patch palette — change them there, not by hunting for new art. The discrete debris/barrier sprites from Kenney's Racing Pack used before the rough-terrain/oil-slick rework have been removed (no sourced art fits a ground-level "patch you drive over" shape).
- **Audio** (`public/assets/audio/`): Kenney's Impact Sounds / Sci-fi Sounds / Interface Sounds, loaded via `this.load.audio()` under the same `SFX_KEYS` the previous Web-Audio-synthesis version used, so `GameScene.playSfx()` didn't need to change at all.

Full source list and license notes: `CREDITS.md`. No continuous/looped engine sound; only discrete one-shot effects (collision, explosion, rocket, gunfire, pickup, game over).

### Weapon visuals & in-world meter

The three weapons previously differed only by a faint tint on the same shared bullet sprite, which made the side guns and turret hard to tell apart in motion. `WEAPON_VISUALS` (`config.ts`) now gives each weapon its own tint, projectile scale, and blend mode — rocket (large, saturated red-orange, `normal` blend, scale 2.4), side guns (pale yellow, `add`, scale 1.3), turret (small, cyan, `add`, scale 0.9) — applied in `GameScene.spawnPlayerProjectile()`. All three scales were bumped up from their original values after playtesting found bullets hard to spot once cars (and so the overall scene) shrank to `CAR_SCALE`.

The rocket specifically needed `normal` blend instead of `add`: its default firing lane is dead center on the road (`player.x` when not steering), which is exactly where the tan road-divider dashes scroll past, and an additively-blended light-colored glow washes out to near-white — i.e. effectively invisible — every time it crosses one. Since the rocket fires rarely (long cooldown, 6 shots) each shot disappearing like that reads as "the weapon doesn't work," unlike the rapid-fire side guns/turret where one faint frame is lost in the stream. `normal` blend with a saturated tint keeps the rocket visibly distinct from both the dark road and the pale divider.

`GameScene.drawWeaponMeter()` draws a small `Graphics` overlay positioned just above the player car each frame (`WEAPON_METER.offsetY` above `player.displayHeight / 2`), shape depending on the equipped weapon:

- **Rocket**: a filling bar for reload progress. An earlier version added a "RELOAD" text label here because the bar alone read as ambiguous — easily mistaken for a health bar, especially once health bars were added in the same red/green color family. The label was removed again once the health bars moved *below* the car instead (see Health bars and high score) — with the two no longer sharing the same on-screen real estate above the car, the ambiguity that motivated the label in the first place is gone.
- **Side guns**: two mirrored dots, each sweeping its own arc centered outward at `±SIDE_GUN_MOUNTS.centerAngleDeg` (90°) rather than around 0°/forward, positioned at each side mount (`±sideGunMountOffsetX()` from center). Both markers derive from the one shared `sweepAngleDeg` via `GameScene.sideGunAngleDeg(side, sweepAngleDeg)` — `-90 + sweep` for the left mount, `90 - sweep` for the right — so the meter and the actual firing angle in `spawnPlayerProjectile` always agree. This direction flip (outward instead of forward) followed directly from playtesting: mounting the barrels at the sides but still sweeping them around a forward-centered arc made the weapon *look* like it was still firing forward in a cluster, not reading as side-mounted at all. `SIDE_GUN_SWEEP.maxAngleDeg` was also narrowed (55° → 40°) after playtesting found the original arc swung too far back toward the rear at its extremes.
- **Turret**: the existing widening/narrowing spread cone, plus a crosshair (`strokeCircle` + cross lines) drawn at `this.input.activePointer`'s world position — added because the turret's aim direction otherwise had no visible indicator at all beyond the spread cone above the car, which doesn't show *where* on screen the pointer actually is.

This is a glanceable supplement to the existing numeric HUD text (`aimText`), not a replacement — the corner text stays the precise readout. The meter and the firing-angle math in `spawnPlayerProjectile`/`computeTurretAimDeg` share one canvas-angle convention (0° = straight ahead/up, positive = clockwise) via `GameScene.aimPointOnArc()`, so the marker position always matches where a shot fired at that instant would actually go.

Side guns fire from two side mounts instead of one front-center point (`GameScene.spawnPlayerProjectile`/`sideGunMountOffsetX()`, offset by the car's own half-width plus `SIDE_GUN_MOUNTS.extraOffsetPx` so the barrels visibly poke out past the body) — a single trigger pull spawns one projectile from each mount, aimed via `sideGunAngleDeg` (mirrored outward, see above), consuming the usual one ammo per pull (so the twin shot is "free" relative to the old single-shot ammo cost; `WEAPONS.sideguns.damage` was lowered from 18 to 10 per projectile to compensate for firing two at once).

### Shared weapon instability

Originally only the turret's spread reacted to handling (steering/off-road/drift). Per gameplay.md, all three weapons should get shakier together at speed and on rough terrain too, representing the car itself jolting the driver around rather than a per-weapon cause. `WeaponController.update()` (`weapons.ts`) computes one `sharedInstabilityDeg` value from `WeaponUpdateParams.speedFraction`/`onRoughTerrain` (both optional, defaulting to 0/`false` so existing callers/tests that don't pass them are unaffected) via `WEAPON_INSTABILITY` in `config.ts`, then folds it into all three weapons' existing mechanics rather than introducing a fourth parallel system:

- **Turret**: added directly into `targetSpread` alongside the existing steering/off-road additions, before the drift multiplier and smoothing — so it's just one more contributor to the spread the turret already had.
- **Side guns**: added as an unconditional jitter term on `sweepAngleDeg` (separate `deterministicJitter` call/seed offset from the existing steering-triggered jitter, so the two don't cancel out when both are active), scaled by `dtSeconds` like the existing jitter.
- **Rocket**: a wholly new `rocketSpreadDeg` state field (smoothed toward `sharedInstabilityDeg` the same way the turret smooths toward its own target, via `WEAPON_INSTABILITY.smoothing`), since the rocket previously had no instability mechanic of its own — `tryFire()` now rolls a `pseudoRandom`-based spread offset for it exactly like the turret already did, deflecting it off dead-ahead.

`PlayerCar.drive()` computes `speedFraction` (`|forwardSpeed| / PLAYER_HANDLING.maxForwardSpeed`, clamped to 1) and passes through `input.onRoughTerrain` into `weapons.update()` each frame, alongside the pre-existing steering/offRoad/drifting params.

## Health bars and high score

`GameScene.drawHealthBars()` draws a small `Graphics` bar (`HEALTH_BAR` in `config.ts`) below the player (green) and above every active enemy (red), proportional to `health / maxHealth` — a graphical supplement to the numeric health readout, following the same pattern as the weapon meter. Drawn fresh every frame (`g.clear()` then redraw) rather than as persistent per-entity game objects, since enemies are pooled/recycled constantly and a single shared `Graphics` avoids having to manage a parallel pool of bar objects.

The player's bar specifically sits *below* the car rather than above, asymmetric with the enemy bars: the area above the player is already used by `drawWeaponMeter()` for the equipped weapon's aim/readiness indicator (see Weapon visuals & in-world meter), and an earlier version that put both there at once made the rocket's reload bar genuinely hard to distinguish from a health bar (same general shape, overlapping color range) — moving the player's health bar below the car instead frees up the full space above for the weapon meter and removed the need for the "RELOAD" label that had been added to work around the ambiguity. Enemies have no weapon meter of their own, so their bars stay above with no such conflict.

## Weapon sidebar

`WEAPON_SIDEBAR` (`config.ts`) positions a persistent sidebar at `x: 406` — just past `ROAD_X + ROAD_WIDTH` (400), i.e. inside the dead-ground margin to the road's right, the one stretch of the 480-wide canvas with nothing else drawn on it. `GameScene.create()` builds one row per `WEAPON_IDS` entry: a static `Rectangle` swatch tinted with that weapon's `WEAPON_VISUALS` color (created once — the tint never changes, so no need to redraw it every frame), a number-key label, a name label (`WEAPON_LABELS`), and an ammo `Text` kept in `sidebarAmmoTexts` (keyed by `WeaponId`) so `updateWeaponSidebar()` can update just the ammo count each frame without touching the static parts. A separate `Graphics` (`weaponSidebarHighlight`, depth `DEPTHS.hud - 1` so it sits behind the row's text/swatch) is cleared and redrawn each frame as a translucent rect behind whichever row matches `player.weapons.current`, by index into `WEAPON_IDS` — the same array order the existing `keydown-ONE/TWO/THREE` bindings use, so the sidebar's numbering always matches what the keys actually do.

This exists alongside, not instead of, the numeric `weaponText` HUD line in the corner — the sidebar's job is showing the *full roster* and how to reach each one at a glance, which a single "Weapon: X (ammo)" line can't do on its own.

Best distance is persisted across runs via `localStorage` (`HIGH_SCORE_STORAGE_KEY` in `GameScene.ts`) — read once in `create()`, shown immediately on the HUD, and compared/written in `endGame()`. The game-over overlay shows either the stored best or "New Best!" depending on whether the just-finished run beat it. This is the one piece of state that survives a `scene.restart()` (everything else resets in `create()`), and intentionally so — it's a record to chase, not meta-progression (see gameplay.md's Scoring & progression).

## Input

- Movement: keyboard only — WASD and arrow keys both work (W/Up accelerate, S/Down brake-then-reverse, A/Left and D/Right steer), plus Shift for drift, polled continuously each frame via `Key.isDown` rather than discrete keydown events, since these are rate controls, not one-shot toggles.
- Weapon select: keyboard, one-shot `keydown-ONE/TWO/THREE` listeners switching `PlayerCar.selectWeapon()`.
- Fire: weapon-dependent, per gameplay.md — Space (polled `isDown`) for rocket/side-guns, pointer `isDown` (click-and-hold) for the turret, since the turret's aim already tracks the pointer position every frame regardless of fire state.
- Restart: keyboard (Space) and pointer (tap/click) — dual-path as CLAUDE.md requires.
- **Touch parity is explicitly deferred for now**, for movement *and* for the turret's mouse-aim. There's no touch equivalent of WASD accelerate/steer or pointer-aim yet, and no scheme is designed for either. This is a deliberate, called-out exception to CLAUDE.md's "new controls must work through both keyboard and touch/pointer paths" rule, not a silent gap — revisit when touch parity is prioritized.

## Testing

Vitest is wired up (`npm test`), 34 tests across three files, all pure-logic:

- `playerPhysics.test.ts` — handling math: accel/brake/reverse, speed-scaled steering, off-road drag, drift (speed drain + turn multiplier + minimum-speed gating), the temporary max-speed override used by the speed-boost pickup, and the two terrain effects (extra drag on rough terrain; reduced turn authority + a fixed-direction drift bias while oil-slicked, including that a zero/flipped bias contributes nothing/the opposite push).
- `weapons.test.ts` — `WeaponController`: ammo consumption/cooldown gating, ammo refill clamped to max, side-gun sweep bounds, turret instability rising with steering/off-road/drift, and the shared speed/rough-terrain instability reaching the turret, rocket, and side-gun sweep alike.
- `enemyBehaviors.test.ts` — `computeLateralVelocity`: chase-toward-player, keep-distance-then-drift-toward (Shooter), no-lateral-movement (Heavy), avoidance ignored beyond the avoidance radius, a passive archetype's avoidance pulling it away from a close rival, an aggressive archetype only weakly avoiding, and Heavy's zero lateral speed meaning it can't avoid at all either; plus `smoothLateralVelocity`'s easing-toward-target behavior.
- `playerPhysics.test.ts` predates the other two; new pure modules followed the same pattern immediately rather than being left untested.

Phaser scene wiring, rendering, and input plumbing stay covered by manual `npm run dev` playtesting per CLAUDE.md — not worth fighting Phaser's runtime in a unit test for that layer.
